## 1. 安装部署手册

#### 1. 准备安装包

- openresty-edge-xxx.tar.gz：

  ```sh
  install-edge.sh
  openresty-1.29.2.5edge.tar.gz
  openssl-3.5.5.tar.gz
  pcre2-10.47.tar.gz
  zlib-1.3.1.tar.gz
  ```

#### 2. 安装 openresty-edge 版本

- 通过脚本 `install-edge.sh` 进行安装

- 命令:

  ```sh
  install-edge.sh {openresty_dir}
  ```

- 例：这里安装到 /usr/local/openresty/test

  ```sh
  ./install-edge.sh /usr/local/openresty/test
  ```

#### 3. 部署 edge 服务

- 通过管理命令 `{openrety-dir}/bin/manager` 进行操作

- 命令:

  ```sh
  {openrety-dir}/bin/manager install {edge-new-dir}
  ```

- 例：在当前目录下部署2个 edge 服务，uap-edge1 和 uap-edge2

  ```sh
  [root@2e8b07d41937 uapm]# /usr/local/openresty/test/bin/manager install uap-edge1
  /usr/local/openresty/test
  init OK.
  install edge ok.
  
  [root@2e8b07d41937 uapm]# /usr/local/openresty/test/bin/manager install uap-edge2
  /usr/local/openresty/test
  init OK.
  install edge ok.
  ```

#### 4. 升级 edge 服务

- 通过管理命令 `{openrety-dir}/bin/manager` 升级已经部署好的 `edge` 服务

- 命令:
  ```sh
  {openrety-dir}/bin/manager upgrade {edge-old-dir}
  ```

  对使用 bin/edge 管理的 edge 服务，可以通过该命令进行升级，
  升级前需要停止当前服务，升级过程也会尝试停止当前服务

- 例如：升级当前目录下的 uap-edge 服务

  ```sh
  [root@2e8b07d41937 uapm]# /usr/local/openresty/test/bin/manager upgrade uap-edge
  stopping edge.
  nginx: [error] open() "/tmp/uapm/uap-edge/logs/nginx.pid" failed (2: No such file or directory)
  /usr/local/openresty/test
  init OK.
  upgrade edge ok.
  ```

  或 进入指定目录，升级当前目录

  ```sh
  [root@2e8b07d41937 uapm]# cd uap-edge
  [root@2e8b07d41937 uap-edge]# /usr/local/openresty/test/bin/manager upgrade .
  stopping edge.
  nginx: [error] open() "/tmp/uapm/uap-edge/logs/nginx.pid" failed (2: No such file or directory)
  /usr/local/openresty/test
  init OK.
  upgrade edge ok.
  ```

- 查看升级后 edge 信息
  ```sh
  [root@2e8b07d41937 uap-edge]# bin/edge
    version: 1.3.0.26020812 (edge-2.7.5.26012617) [x86_64]
    
    usage: edge {init|start|stop|reload|test} [openresty_path]
    eg:
      bin/edge init /path/to/openresty  (if openresty dir was changed)
      bin/edge init                     (if edge.env was modified)
      bin/edge init-{x86/arm}
      bin/edge cron-{init/clear/update}
      bin/edge pack-{add/list/rebase} [ver]
      bin/edge start/stop/reload/test
    
    inited openresty dir: /usr/local/openresty/test, ver: openresty/1.27.1.2
  [*]2.7.5.26012617
  ```

## 2. 配置管理手册

### 1. 管理 edge 服务

###### 1.1 关联 openresty 目录

- 启动服务前，必须关联上已经安装好的 openresty 目录

  ```sh
  bin/edge init {openresty_dir}
  ```

###### 1.2 初始化 edge 服务

- 如果 edge 服务有任何调整，需要重新初始化

  ```sh
  bin/edge init
  ```

###### 1.3 启停 edge 服务

- 通过下面命令启动、停止、reload 服务

  ```sh
  bin/edge start
  bin/edge stop
  bin/edge reload
  ```

> **注意**：如果启动时遇到类似报错：`[emerg] bind() to unix:....sock failed (48: Address already in use)`，可以先执行 `bin/edge stop`，再重新启动即可。

###### 1.4 切换 edge 版本

  当 openresty-edge 中存在多个 edge 版本包时，可以按如下步骤切换 edge 版本

- 1.4.1 查看包列表

  ```sh
  [root@2e8b07d41937 uap-edge]# bin/edge pack-list
  [*]2.7.5.26012617
  2.7.6.26020421
  ```

- 1.4.2 修改 edge.env 相关配置

  配置服务端口及管理端口，新增或修改 deploy.edge-pack 的版本，等

  ```yaml
  deploy:
    ...
    edge-pack: 2.7.6.26020421

    http:
      edge:
        listen:
          - addr: 0.0.0.0:9980
      admin:
        listen:
          - addr: 0.0.0.0:9990
    ...
  ```

- 1.4.3 初始化 edge 服务

  ```sh
  [root@2e8b07d41937 uap-edge]# bin/edge init
  init OK.
  ```

- 1.4.4 查看 edge 服务版本

  ```sh
  [root@2e8b07d41937 uap-edge]# bin/edge
    version: 1.3.0.26020812 (edge-2.7.6.26020421) [x86_64]
    
    usage: edge {init|start|stop|reload|test} [openresty_path]
    eg:
      bin/edge init /path/to/openresty  (if openresty dir was changed)
      bin/edge init                     (if edge.env was modified)
      bin/edge init-{x86/arm}
      bin/edge cron-{init/clear/update}
      bin/edge pack-{add/list/rebase} [ver]
      bin/edge start/stop/reload/test
    
    inited openresty dir: /usr/local/openresty/test, ver: openresty/1.27.1.2
  [*]2.7.5.26012617
  2.7.6.26020421
  ```

  最后显示库里有2个版本，其中，
      version: 1.3.0.26020812 (edge-2.7.6.26020421) [x86_64]
  表示，当前实例使用的 (edge-2.7.6.26020421)

- 1.4.5 启动 edge 服务

  ```sh
  [root@2e8b07d41937 uap-edge]# bin/edge start
  ```

  注意：如果 deploy.edge-pack 指定的版本不存在，不会生效


### 2. 管理 openresty-edge 版本包

- 通过管理命令 `{openrety-dir}/bin/manager` 管理 `edge` 版本包，

  也可以通过已关联上该 openresty 的 edge 服务，使用命令 `bin/edge` 进行操作

  ```sh
  pack-list    显示当前包列表
  pack-add     新增一个edge包
  pack-rebase  合并指定包到基础包中
  ```

  `edge` 版本包：按下面目录结构，将目录 `edge-pack-{版本号}` 打包成 `edge-pack-{版本号}.tar.gz`，{版本号}也可以是其它别名，如：1、2、3等

  ```sh
  edge-pack-{版本号}
  └──edge
  ```

- 命令:

  ```sh
  {openrety-dir}/bin/manager pack-list
  {openrety-dir}/bin/manager pack-add edge-pack-{ver}.tar.gz
  {openrety-dir}/bin/manager pack-rebase {ver}
  ```

  或:

  ```sh
  bin/edge pack-list
  bin/edge pack-add edge-pack-{ver}.tar.gz
  bin/edge pack-rebase {ver}
  ```

- 例：


### 3. edge.env 配置项详解

`edge.env` 是网关部署的核心配置文件，各层级配置项的详细说明如下：

#### 3.1 `deploy` 字段下配置项

| 指令 | 默认值 | 说明 |
| :--- | :--- | :--- |
| `prefix` | `edge` | 配置名，存储配置的名称，修改后需要重新导入路由等配置。 |
| `worksite` | `edge_app` | 业务名，预处理时，会自动调用该业务同名的 lua 模块。 |
| `edge-pack` | — | 多版本切换，可以在这指定要使用的 edge 包版本。 |
| `user` | - | worker 进程运行用户。 |
| `master_process` | `on` | 是否启用 master/worker 模型。 |
| `worker_processes` | `auto` | 工作进程数。 |
| `worker_cpu_affinity` | auto | 绑定 worker 进程到指定 CPU 核，减少上下文切换。 |
| `worker_rlimit_nofile` | 65535 | 每个 worker 最大打开文件数。 |
| `error_log` | `logs/error.log` | 错误日志路径。 |
| `error_log_level` | `warn` | 错误日志级别(debug/info/notice/warn/error/crit)。 |
| `pid` | `logs/nginx.pid` | 主进程 PID 文件路径。 |

---

#### 3.1.1 `deploy.events` 字段下配置项

| 指令 | 默认值 | 说明 |
| :--- | :--- | :--- |
| `use` | `epoll` | I/O 多路复用方式，Linux 推荐 `epoll`。 |
| `worker_connections` | `65535` | 每个 worker 最大并发连接数。 |
| `multi_accept` | `off` | 开启后 worker 一次接受所有新连接。 |
| `accept_mutex` | `off` | 序列化 accept，避免多 worker 惊群。 |
| `accept_mutex_delay` | `500ms` | `accept_mutex` 开启时的等待时间。 |

---

#### 3.1.2 `deploy.http` 字段下配置项

| 指令 | 默认值 | 说明 |
| :--- | :--- | :--- |
| `sendfile` | `on` | 零拷贝传输静态文件。 |
| `tcp_nopush` | `on` | `sendfile` 开启时合并响应头与文件到一个包。 |
| `tcp_nodelay` | `on` | 长连接时禁用 Nagle，减少延迟。 |
| `server_tokens` | `off` | 是否在响应头中暴露 nginx 版本。 |
| `keepalive_timeout` | `60s` | keep-alive 连接空闲超时，设 `0` 关闭长连接。 |
| `keepalive_requests` | `100000` | 单个长连接最多处理的请求数。 |
| `client_header_timeout` | `30s` | 读取请求头超时。 |
| `client_body_timeout` | `30s` | 读取请求体超时（两次读之间）。 |
| `send_timeout` | `10s` | 向客户端发送响应超时（两次写之间）。 |
| `client_header_buffer_size` | `4k` | 请求头内存缓冲大小。 |
| `large_client_header_buffers` | `4 128k` | 大请求头的缓冲数量和单个大小。 |
| `client_max_body_size` | `32m` | 请求体最大限制，上传文件时需调大，设 `0` 不限。 |
| `client_body_buffer_size` | `8m` | 请求体内存缓冲，超出后写临时文件。 |
| `proxy_buffer_size` | `32k` | 读取上游响应头的缓冲大小。 |
| `proxy_buffers` | `16 64k` | 响应体缓冲数量和大小。 |
| `gzip` | `on` | 开启 gzip 压缩。 |
| `gzip_comp_level` | `4` | 压缩级别 1-9，推荐 4-6 平衡 CPU 与压缩率。 |
| `gzip_min_length` | `1024` | 最小压缩响应大小（字节）。 |
| `gzip_proxied` | `off` | 对代理请求的压缩策略。 |
| `gzip_types` | `text/plain text/css text/javascript application/javascript application/x-javascript application/xml application/json` | 需要压缩的 MIME 类型。 |
| `gzip_vary` | `off` | 添加 Vary: Accept-Encoding 响应头。 |
| `access_log` | `off` | 访问日志路径及格式，如：logs/access.log main buffer=16384 flush=3。 |
| `log_format main` | `escape=default '$remote_addr $http_cdn_src_ip - $remote_user [$time_local] $http_host "$request" $status $body_bytes_sent $request_time "$content_type" $content_length "$http_referer" "$http_user_agent" "$http_cookie" "$http_x_forwarded_for" "$connection" "$connection_requests" "$upstream_addr" "$upstream_status" "$upstream_response_time" $request_time'` | 自定义日志格式，可引用 `$变量`。 |

#### 3.1.2.1 `deploy.http.edge` 字段下配置项

| 指令 | 默认值 | 说明 |
| :--- | :--- | :--- |
| `listen` | `0.0.0.0:9980` | 服务端口，可以配置多个监听端口/地址。 |
| `server_name` | `_` | 虚拟主机域名。 |
| `client_header_buffer_size` | `4k` | 请求头内存缓冲大小。 |
| `large_client_header_buffers` | `4 128k` | 大请求头的缓冲数量和单个大小。 |
| `client_max_body_size` | `32m` | 请求体最大限制，上传文件时需调大，设 `0` 不限。 |
| `client_body_buffer_size` | `8m` | 请求体内存缓冲，超出后写临时文件。 |
| `underscores_in_headers` | `off` | 是否接收带下划线的 Header。 |
| `proxy_buffer_size` | `32k` | 读取上游响应头的缓冲大小。 |
| `proxy_buffers` | `16 64k` | 响应体缓冲数量和大小。 |
| `ssl_certificate` | cert/server.crt | 证书文件路径（PEM 格式）。 |
| `ssl_certificate_key` | cert/server.key | 私钥文件路径. |

### 4. `node.env` 差异化配置说明

`node.env` 是可选的节点差异化配置文件，位于网关服务的根目录下（与 `edge.env` 同级）。

- **功能与格式**：其配置格式与语法规则完全与 `edge.env` 相同。
- **覆盖机制**：在执行初始化命令（如 `bin/edge init`）加载配置时，网关会合并这两个文件，且 `node.env` 中配置的属性会**覆盖** `edge.env` 中相同的属性。
- **使用场景**：适用于部分节点与全局默认配置不一致的情况。例如：
  - 某个高配节点请求量是其他节点的多倍，需要同时开启多个服务端口，且该节点上的限流插件 `traffic_limit_count` 本地限制次数（`local_ratio`）也应当调大数倍。
  - 该节点上的 `node.env` 可以配置如下：
    ```yaml
    deploy:
      http:
        edge:
          listen:
            - addr: 0.0.0.0:9980
            - addr: 0.0.0.0:9981
    plugin_attr:
      traffic_limit_count:
        local_ratio: 4
    ```

#### 加解密密钥

下面是加解密默认密钥，可以在 `edge.env` 中进行设置，格式如下：

```yaml
edge:
  admin_encrypt_salt: <SECRET_KEY>
  admin_decrypt_salt: <SECRET_KEY>
```

> 管理接口通信需要进行加解密处理。


##### 加密流程

管理端调用接口发送请求时，请求体应先加密后发给服务器：

1.  首先对请求体做 **sm4** 加密，模式为 `ecb + pkcs7` 填充算法，密钥对应上面配置的 `admin_decrypt_salt`。
2.  然后对加密后的数据做 **base64** 处理。
3.  再把这个数据作为请求体发给服务器。

> [!WARNING]
> 如果不加密，服务器会报错：`invalid request body: Expected value but found T_END at character 1`。

例如：
```bash
echo '{"desc": "global common", "plugins": {"traceid": {}}}' | openssl enc -sm4-ecb -K $(echo -n <SECRET_KEY> | xxd -p) | base64 | tr '+/' '-_' | tr -d =
```


##### 解密流程

服务器正常响应后，应对响应体解密再使用：

1.  管理端收到响应后，先判断响应结果是否为 200, 201 等，否则检查请求参数是否有误。
2.  然后对正常的响应体做 **debase64** 处理。
3.  再对这个数据做 **sm4** 解密，模式为 `ecb + pkcs7` 填充算法，密钥对应上面配置的 `admin_decrypt_salt`。
4.  解密后就是可以正常使用的数据。

例如：
```bash
echo 'opQsb_7JCqmGeYAes532inDYEVE5qJMpfTfqXE4xUcSwd7xcUUCTmwpSEoZLqDMyhOTiVgOmizz9Bhf0gxAW_NhhUmGyP2QG-OYxqTmnOREnj15kbo34OJwJGGai9PllQ9ysy9E5XbJdhRoIrwRtBA' | tr '_-' '/+' | (read tk; len=${#tk}; pad=$(( (4-len%4)%4 )); printf "%s" "$tk"; for i in $(seq 1 $pad); do printf "="; done; echo "") | base64 -d | openssl enc -sm4-ecb -d -K $(echo -n <SECRET_KEY> | xxd -p)
```

##### 路由管理 (Routes)

路由分为 **HTTP 路由** 和 **Stream 路由**（L4 代理）。路由通过定义匹配规则来决定如何处理请求，并将其转发给指定的 Upstream 或 Service。

###### 请求方法

####### HTTP 模式

| 方法 | 路径 | 说明 |
| :--- | :--- | :--- |
| **GET** | `/edge/admin/routes` | 获取 HTTP 路由列表 |
| **GET** | `/edge/admin/routes/{id}` | 根据 ID 获取 HTTP 路由资源 |
| **PUT** | `/edge/admin/routes/{id}` | 根据 ID 创建或替换 HTTP 路由资源 |
| **DELETE** | `/edge/admin/routes/{id}` | 删除 HTTP 路由资源 |

####### Stream 模式

> [!NOTE]
> 如果开启了 `stream` 模式，`stream` 接口管理路径需要加上 `/stream` 前缀。

| 方法 | 路径 | 说明 |
| :--- | :--- | :--- |
| **GET** | `/stream/edge/admin/routes` | 获取 Stream 路由列表 |
| **GET** | `/stream/edge/admin/routes/{id}` | 根据 ID 获取 Stream 路由资源 |
| **PUT** | `/stream/edge/admin/routes/{id}` | 根据 ID 创建或替换 Stream 路由资源 |
| **DELETE** | `/stream/edge/admin/routes/{id}` | 删除 Stream 路由资源 |

###### 请求参数 (Body)

####### 1. HTTP 路由参数

######## 匹配条件

| 参数名 | 类型 | 默认值 | 必填/互斥关系 | 说明 |
| :--- | :--- | :--- | :--- | :--- |
| **uri** | String | 无 | 与 `uris` 二选一 | 匹配的单个请求路径，支持前缀匹配 (如 `/hello/*`)。长度 1 ~ 4096。 |
| **uris** | Array | 无 | 与 `uri` 二选一 | 匹配的请求路径列表，元素为 String。最少 1 项，需唯一。 |
| **host** | String | 无 | 与 `hosts` 互斥 | 匹配的单个域名，支持通配符 (如 `*.bar.com`)。 |
| **hosts** | Array | 无 | 与 `host` 互斥 | 匹配的域名列表，元素为 String。最少 1 项，需唯一。 |
| **remote_addr** | String | 无 | 与 `remote_addrs` 互斥 | 匹配的单个客户端 IP 或 CIDR (支持 IPv4/IPv6)。 |
| **remote_addrs** | Array | 无 | 与 `remote_addr` 互斥 | 匹配的客户端 IP 或 CIDR 列表，元素为 String。最少 1 项，需唯一。 |
| **methods** | Array | 无 | 可选 | 匹配的 HTTP 方法列表 (如 `["GET", "POST"]`)，需唯一。 |
| **priority** | Integer | `0` | 可选 | 匹配优先级，数值越大优先级越高。 |
| **vars** | Array | 无 | 可选 | 复杂的变量匹配规则。详见 [Edge 表达式列表](#8-核心表达式引擎)。 |

| 参数名 | 类型 | 默认值 | 说明 |
| :--- | :--- | :--- | :--- |
| **upstream** | Object | 无 | 内联 Upstream 配置对象。详见 [Upstream 配置](#上游-upstreams)。 |
| **upstream_id** | String \| Integer | 无 | 引用的外部 Upstream ID。 |
| **service_id** | String \| Integer | 无 | 引用的 Service ID。 |

######## 插件配置

| 参数名 | 类型 | 默认值 | 必填/互斥关系 | 说明 |
| :--- | :--- | :--- | :--- | :--- |
| **plugins** | Object | 无 | 可选 | 插件配置对象，键为插件名称，值为对应插件的配置。 |
| **plugin_config_id** | String \| Integer | 无 | 与 `plugin_config_ids` 互斥 | 引用已定义的单个插件组配置 ID。 |
| **plugin_config_ids**| Array | 无 | 与 `plugin_config_id` 互斥 | 引用已定义的插件组配置 ID 数组，元素为 String 或 Integer。最少 1 项，需唯一。 |

> [!NOTE]
> **插件合并与执行优先级**：
> - 执行时会合并 `plugins`（直接配置在路由上的插件）与引用的插件组（通过 `plugin_config_id` 或 `plugin_config_ids` 指定）中所有满足条件的插件。
> - 若合并过程中存在同名插件，其生效优先级为：
>   1. **第一优先级**：路由本身直接定义的 `plugins`。
>   2. **第二优先级**：引用的插件组列表（按 `plugin_config_ids` 数组顺序，排在前面的插件组优先）。
>   3. **第三优先级**：插件组内部（按插件组内部的优先级：`var_plugins` 中满足条件的插件优先于默认 `plugins`）。

| 参数名 | 类型 | 默认值 | 说明 |
| :--- | :--- | :--- | :--- |

> - 执行时会合并 `plugins`（直接配置在路由上的插件）与引用的插件组（通过 `plugin_config_id` 或 `plugin_config_ids` 指定）中所有满足条件的插件。

>   3. **第三优先级**：插件组内部（按插件组内部的优先级：`var_plugins` 中满足条件的插件优先于默认 `plugins`）。

##### 上游管理 (Upstreams)

Upstream 用于对多个服务节点进行负载均衡。

###### 请求参数 (Body)

####### 1. 负载均衡

| 参数名 | 类型 | 默认值 | 必填/互斥关系 | 说明 |
| :--- | :--- | :--- | :--- | :--- |
| **type** | String | `"roundrobin"` | 可选 | 负载均衡算法。支持的值有：`roundrobin` (轮询), `chash` (一致性哈希), `ewma` (最小延迟), `least_conn` (最少连接)。 |
| **hash_on** | String | `"vars_combinations"` | 可选 | 一致性哈希算法的 Hash 基准。可选值：`vars`, `header`, `cookie`, `vars_combinations`。 |
| **key** | String | 无 | 可选 | 一致性哈希算法的哈希键。配合 `hash_on` 使用：<br>1. 当 `hash_on` 指定为 `header` 时，表示请求头字段；<br>2. 当 `hash_on` 指定为 `cookie` 时，表示 cookie 中的键值（区分大小写）；<br>3. 当 `hash_on` 指定为 `vars` 时，表示 edge 变量，其中 `http_name`、`cookie_name` 即对应上述请求头和 cookie；<br>4. 当 `hash_on` 为默认值 `vars_combinations` 时，表示 vars 中变量的组合，需使用 `${}` 引用变量（如 `${http_name}`、`${cookie_name}`），且支持如 `node_${arg_name}_${remote_addr}` 的自定义组合。 |
| **default_key** | String | 无 | 可选 | 一致性哈希算法的默认哈希键。当 `key` 指定的值不存在时，其工作机制如下：<br>1. 若设置了该值，会取该设定键对应的值；<br>2. 若 `hash_on` 为 `cookie`，即使没有设置该值，也会去取 `traceid` 的值；<br>3. 以上结果仍未获取到值时，默认会取 `remote_addr` 的值，并在日志中给出警告信息 `"chash_key fetch is nil, use default chash_key[remote_addr]"`。 |

### 1. 插件的复用功能（别名调用）

为了支持在单个路由中多次执行同一个插件的不同逻辑，网关支持了**插件的复用功能**。

#### 1.1 使用规则
- **可复用的插件**（未标注为 `unique` 的插件）可以在一个路由中通过别名的方式多次调用。
- **命名规范**：别名使用破折号 `-` 进行连接，格式为：`{原插件名}-{别名}`。每个主插件可以拥有多个不同的别名。
- **声明与注册**：别名插件依赖主插件的存在，且必须在 `edge.env` 的 `ex_plugins`（或 `ex_stream_plugins`）配置中进行声明注册。
- **独立属性**：所有声明的别名插件都可以像普通插件一样，在路由中配置自己独立的数据属性（Metadata Config / Config）。

#### 1.2 注册示例
在部署配置 `edge.env` 中注册别名插件：
```yaml
ex_plugins:
  plugin_demo-test1: true      # 启用别名插件 1，使用默认优先级
  plugin_demo-test2: 1999      # 启用别名插件 2，并自定义其执行优先级为 1999
```

> [!WARNING]
> 凡是标注为 `unique` 的插件（即不可复用的插件，如 `cors`、`proxy_rewrite` 等），**不支持**使用别名进行多次实例化配置。

### 监控指标列表

以下为网关收集并导出的核心监控指标：

#### HTTP 阶段监控指标
- `edge_bandwidth`：服务消耗的物理总流量大小（单位：字节）。可以通过 `type="ingress"`（入流量）和 `type="egress"`（出流量）区分流量方向。
- `edge_http_requests_total`：自网关启动以来累计处理的客户端 HTTP 请求总量。
- `edge_http_status`：响应状态码统计，可通过 `code` 标签（如 `200`、`404`、`502` 等）对各路由和 matched_uri 的命中频次进行详细观测。
- `edge_http_latency`：HTTP 请求各类耗时的分布直方图（单位：毫秒）。可通过 `type` 区分总耗时（`request`）、后端连接耗时（`upstream`）及网关自身开销（`edge`）等。导出的直方图指标包含以下三项：
  - `edge_http_latency_bucket`：耗时计数，统计 `10, 20, 50, 100, 200, 500, 1000, 2000, 5000, 10000ms` 以及大于 10000ms 的请求耗时分布数。
  - `edge_http_latency_count`：各耗时来源的请求计数。
  - `edge_http_latency_sum`：各耗时来源的累计总请求耗时。
- `edge_nginx_http_current_connections`：网关底层 Nginx 的当前各种连接状态数据统计（如活跃连接、等待连接等）。
- `edge_shared_dict_capacity_bytes` / `edge_shared_dict_free_space_bytes`：网关内部各个共享内存字典的总体容量与剩余空闲空间监控。
- `edge_upstream_status`：各个后端上游节点健康检查状态的存活率指标。
- `edge_plugin_risk` / `edge_plugin_riskid`：网关各防守安全插件的累计拦截统计，支持下钻细化到具体的 WAF 规则 ID。
- `edge_plugin_latency`：各具体业务逻辑插件在处理链中的实际执行开销耗时分布直方图。

#### L4 Stream 阶段监控指标
- `edge_stream_connection_total`：每个 L4 代理路由的累计连接次数计数。

### 2. 仪表盘面板行 (Rows) & 面板 (Panels) 详解

---

#### Row: General (总体运行状况)

展示全局的运行流量、耗时、状态码分布和 WAF 安全规则命中大盘情况。

##### 1. requests(tps) (ID: 27)
* **面板类型**：Time Series (时间序列图)
* **说明**：展示系统的每秒请求速率（TPS），可分别从 Job、Instance 维度观测整体吞吐量。
* **PromQL 查询语句**：
  ```promql
  sum by(job)(rate(${edge_prefix}http_status{job=~"$edge_job", instance=~"$edge_instance", route!~"$route_rtengine"}[$__rate_interval])) * $show_interval
  ```
  ```promql
  sum by(instance) (rate(${edge_prefix}http_status{job=~"$edge_job", instance=~"$edge_instance", route!~"$route_rtengine"}[$__rate_interval])) * $show_interval
  ```
* **截图参考**：
  ![总体 TPS 请求吞吐率]

##### 2. latency_avg(ms) (ID: 28)
* **面板类型**：Time Series
* **说明**：分阶段（`edge` 自身延迟、`upstream` 后端响应延迟、`request` 客户端总耗时）展示 HTTP 请求的平均耗时。
* **PromQL 查询语句**：
  ```promql
  sum by(type) (rate(${edge_prefix}http_latency_sum{job=~"$edge_job", instance=~"$edge_instance", type!~"client_.*", route!~"$route_rtengine"}[$__rate_interval])) / sum by(type) (rate(${edge_prefix}http_latency_count{job=~"$edge_job", instance=~"$edge_instance", type!~"client_.*", route!~"$route_rtengine"}[$__rate_interval]))
  ```
* **截图参考**：
  ![全局平均延迟]

##### 3. status (ID: 26)
* **面板类型**：Time Series
* **说明**：统计系统返回的 HTTP 响应状态码分布，能直观发现 `2xx`、`3xx`、`4xx`、`5xx` 请求速率。
* **PromQL 查询语句**：
  ```promql
  sum by(code) (rate(${edge_prefix}http_status{job=~"$edge_job", instance=~"$edge_instance", route!~"$route_rtengine"}[$__rate_interval])) * $show_interval
  ```
* **截图参考**：
  ![响应状态码分布]

##### 4. latency_tp(ms) (ID: 29)
* **面板类型**：Time Series
* **说明**：客户端总请求时间（type="request"）的 TP50, TP75, TP95, TP99 百分位耗时（通过直方图分桶指标计算）。
* **PromQL 查询语句**：
  - **TP50**:
    ```promql
    histogram_quantile(0.50, sum by(le) (rate(${edge_prefix}http_latency_bucket{job=~"$edge_job", instance=~"$edge_instance", type="request", route!~"$route_rtengine"}[$__rate_interval])*${show_interval}))
    ```
  - **TP95**:
    ```promql
    histogram_quantile(0.95, sum by(le) (rate(${edge_prefix}http_latency_bucket{job=~"$edge_job", instance=~"$edge_instance", type="request", route!~"$route_rtengine"}[$__rate_interval])*${show_interval}))
    ```
* **截图参考**：
  ![客户端耗时分位数]

##### 5. rules(%) (安全规则命中比例) (ID: 30)
* **面板类型**：Time Series
* **说明**：安全规则命中比例。展示 WAF 规则（rules）和安全风险（risks）命中占比整体请求总量的百分比（%）。
* **PromQL 查询语句**：
  - **安全风险标识比例 (Risks)**:
    ```promql
    sum(rate(${edge_prefix}plugin_risk{job=~"$edge_job", instance=~"$edge_instance", route!~"$route_rtengine"}[$__rate_interval]))*100/sum(rate(${edge_prefix}http_status{job=~"$edge_job", instance=~"$edge_instance", route!~"$route_rtengine"}[$__rate_interval]))
    ```
  - **安全规则命中比例 (Rules)**:
    ```promql
    sum(rate(${edge_prefix}plugin_riskid{job=~"$edge_job", instance=~"$edge_instance", route!~"$route_rtengine"}[$__rate_interval]))*100/sum(rate(${edge_prefix}http_status{job=~"$edge_job", instance=~"$edge_instance", route!~"$route_rtengine"}[$__rate_interval]))
    ```
* **截图参考**：
  ![安全规则命中比例]

##### 6. risks(%) (安全风险标识比例) (ID: 31)
* **面板类型**：Time Series
* **说明**：安全风险标识比例。按各防御/安全插件（CORS、Auth、WAF 等）分类统计的具体风险触发占比。
* **PromQL 查询语句**：
  ```promql
  sum by(name) (rate(${edge_prefix}plugin_risk{job=~"$edge_job", instance=~"$edge_instance", route!~"$route_rtengine"}[$__rate_interval])) * $show_interval
  ```
* **截图参考**：
  ![安全风险标识比例]

  ![后端节点 TPS 负载]

##### 4. latency_tp(ms) (ID: 19)

* **说明**：按 Route 路由细分计算的 TP95 请求延迟。
* **PromQL 查询语句**：
  ```promql
  histogram_quantile(0.95, sum by(le,route) (rate(${edge_prefix}http_latency_bucket{job=~"$edge_job", instance=~"$edge_instance", type="request", route!~"$route_rtengine"}[$__rate_interval])*${show_interval}))
  ```
* **截图参考**：
  ![各路由 TP95 分位数耗时]

* **面板类型**：Time Series
* **说明**：统计系统产生 4xx 或 5xx 异常响应状态码时的每秒请求数。
* **PromQL 查询语句**：
  ```promql
  sum without(job, instance, node) (rate(${edge_prefix}http_status{job=~"$edge_job", instance=~"$edge_instance", route!~"$route_rtengine", code=~"[45][0-9]{2}"}[$__rate_interval])) * $show_interval > 0
  ```
* **截图参考**：
  ![错误状态码 TPS 统计]

##### 6. latency_count (ID: 18)
* **面板类型**：Time Series
* **说明**：请求延迟在不同预置时间桶（10ms、100ms、1000ms 等）中的分布与计数。
* **PromQL 查询语句**：
  ```promql
  sum by(le) (rate(${edge_prefix}http_latency_bucket{job=~"$edge_job", instance=~"$edge_instance", type="request", route!~"$route_rtengine", le=~"^(.Inf|10|100|1000)$"}[$__rate_interval]))*${show_interval}
  ```
* **截图参考**：
  ![延迟区间时间桶计数]

##### 7. rules(tps) (ID: 7)
* **面板类型**：Time Series
* **说明**：显示命中安全规则的前 5 名，按照安全插件名称与匹配 URI 聚合统计。
* **PromQL 查询语句**：
  ```promql
  topk(5, sum by(name, matched_uri) (rate(${edge_prefix}plugin_risk{job=~"$edge_job", instance=~"$edge_instance", route!~"$route_rtengine"}[$__rate_interval])) * $show_interval)
  ```

* **说明**：具体 WAF 风险规则 ID（riskid）和命中接口的前 5 名 TPS 统计。
* **PromQL 查询语句**：
  ```promql

  ```
  ```promql
  topk(5, sum by(riskid, matched_uri) (rate(${edge_prefix}plugin_riskid{job=~"$edge_job", instance=~"$edge_instance", route!~"$route_rtengine"}[$__rate_interval])) * $show_interval)
  ```

用户可以自行安装 Grafana + Prometheus 监控 Edge 服务，也可以通过已经部署好的 OpenResty-Edge 中自带的工具进行管理。
- 对于管理节点，可通过 `/path/to/openresty-edge/bin/manager` 进行管理。
- 对于已经部署并绑定了 OpenResty-Edge 的应用实例，同样可以使用 `bin/edge` 进行管理。

**命令调用格式**：
```bash
/path/to/openresty-edge/bin/manager monitor {cmd} [opts]
# 或
bin/edge monitor {cmd} [opts]
```

#### 1. 安装命令 (install)
```bash
bin/edge monitor install [force]
```
- **工作逻辑**：执行时会首先检查当前目录下是否存在 Grafana 和 Prometheus 的离线安装包。如果不存在，脚本将自行联网下载并执行安装。
- **参数说明**：
  - `force`：强制重新安装监控服务。
- **离线安装包下载**（可提前下载对应架构的包放至当前目录以支持无外网环境）：
  - **x86 架构**：
    - Grafana (12.4.3): [grafana_12.4.3_24388279614_linux_amd64.tar.gz](https://dl.grafana.com/grafana/release/12.4.3/grafana_12.4.3_24388279614_linux_amd64.tar.gz)
    - Prometheus (3.5.3): [prometheus-3.5.3.linux-amd64.tar.gz](https://github.com/prometheus/prometheus/releases/download/v3.5.3/prometheus-3.5.3.linux-amd64.tar.gz)
  - **ARM 架构**：
    - Grafana (12.4.3): [grafana_12.4.3_24388279614_linux_arm64.tar.gz](https://dl.grafana.com/grafana/release/12.4.3/grafana_12.4.3_24388279614_linux_arm64.tar.gz)
    - Prometheus (3.5.3): [prometheus-3.5.3.linux-arm64.tar.gz](https://github.com/prometheus/prometheus/releases/download/v3.5.3/prometheus-3.5.3.linux-arm64.tar.gz)

#### 2. 配置命令 (config)
配置需要监控的节点信息，系统默认会配置为 `localhost:9990`。
```bash
bin/edge monitor config [yaml_file]
```
- **不带参数执行**：直接获取现有的监控配置文件内容。
  ```bash
  # bin/edge monitor config

  global:
    scrape_interval: 15s

  scrape_configs:
    - job_name: "edge-waf"
      metrics_path: "/edge/monitor/metrics"
      static_configs:
      - targets:
        - "localhost:9990"
  ```
- **更新配置**：可以先保存已有配置至 YAML 文件，进行编辑修改后重新提交来进行更新。
  ```bash
  # 1. 导出已有配置
  bin/edge monitor config > tmp.yml
  # 2. 编辑配置文件
  vi tmp.yml
  # 3. 提交新配置更新监控服务
  bin/edge monitor config tmp.yml
  ```

#### 3. 清理命令 (clear)
```bash
bin/edge monitor clear
```
- **功能**：停止当前运行的监控服务，并清除所有相关的历史监控数据。

#### 4. 服务启停与状态查看 (start/stop/reload/status)
控制监控组件（Prometheus + Grafana）的生命周期：
- **启动监控服务**：
  ```bash
  bin/edge monitor start [--prometheus-port=9911] [--grafana-port=9922]
  ```
  - **参数说明**：
    - `--prometheus-port=端口号`：指定 Prometheus 的监听端口，默认为 `9911`。
    - `--grafana-port=端口号`：指定 Grafana 的监听端口，默认为 `9922`。
  - **访问说明**：服务启动成功后，可通过浏览器直接访问 **Grafana 监听端口**（默认 `9922`）来使用监控大盘。
  - **内置仪表盘**：首次安装完成后，系统默认配置并导入了两个主要的 Dashboard：
    - `local-edge-xxx`（如 `local-edge-waf`）：即本文档第二部分所详述的实时 WAF 监控仪表盘。
    - `local-edge-m-xxx`（如 `local-edge-m-waf`）：按分钟（Minute）进行数据汇总与统计显示的监控仪表盘。

- **停止监控服务**：
  ```bash
  bin/edge monitor stop
  ```

- **重启/重载监控服务**：
  ```bash
  bin/edge monitor reload [--prometheus-port=9911] [--grafana-port=9922]
  ```
  - 支持在重载时重新指定 Prometheus 和 Grafana 的运行端口。

- **查看服务运行状态**：
  ```bash
  bin/edge monitor status
  ```

##### sub_tcp_port
TCP 子请求配置，通过配置该指令，会自动配置一个 location = /tcp 的转发，通过 --- request 发起 /tcp 的请求， 服务会向指定端口（由 --- sub_tcp_port 指定）再发起一个 tcp 的请求，发送数据优先使用 --- equest 自带请求体，如果没有则由指令 --- sub_tcp_request 指定，默认 "trigger tcp req"，该功能也可通过 --- config 自行定制。

```perl

--- request
POST /tcp
1234567890
```

```perl
location = /t/tcp {

        local sock = ngx.socket.tcp()
        sock:connect("127.0.0.1", 9970)

        local data = sock:receive("*a")
        ngx.say("received: ", data)
    }
}
--- request
GET /t/tcp
```

#### 过滤器

##### 块过滤器

我们可以使用过滤器对数据块（blocks）中的数据进行预处理。

| 过滤器 | 说明 |
|---|---|
| `chop` | 删除字符串最后一个字符（无论是什么字符） |
| `chomp` | 删除字符串末尾的换行符（如果存在） |
| `eval` | 将数据段的值视为一段 Perl 代码并执行 |

例如，我们可以通过指定 `chomp` 过滤器，去掉当前数据段（section）值的最后一个换行符（如果有的话），如下所示：

```perl
--- response_body chomp
Hello world!
```

如果不使用 `chomp` 过滤器，`response_body` 数据段的值将包含一个末尾的换行符。

此外，`eval` 过滤器会将数据段的值视为一段 Perl 代码并执行。例如：

```perl
--- response_body eval
"I don't know what \0 is.\n"
```

## 8. 核心表达式引擎



在路由或插件中配置表达式列表



### 1. 表达式列表


表达式列表是一个或多个节点的数组，其中第一个节点可以是逻辑运算符或表达式或表达式列表，其余节点可以是一个表达式或表达式列表。

每个表达式都是一个包含三个或四个元素的数组，例如:

```json
["left_value", "!(取反运算符，可选)", "operator(比较运算符)", "right_value"]
```



#### 1.1 运算符



##### 1.1.1 比较运算符


| **运算符**       | **描述**                   | **事例**                                                    |
| ---------------- | -------------------------- | ----------------------------------------------------------- |
| `!`              | 否定（取反）               | `["arg_name", "!", "~~", "user[12]"]`                       |
| `==`，`==*`      | 等于                       | `["arg_name", "==", "user"]`                                |
| `!=`，`!=*`      | 不等于                     | `["arg_name", "!=", "user"]`                                |
| `~~`，`~~*`      | 正则匹配                   | `["arg_name", "~~", "user[12]"]`                            |
| `>`              | 大于（数值）               | `["arg_rank", ">", 1]`                                      |
| `>=`             | 大于等于（数值）           | `["arg_rank", ">=", 1]`                                     |
| `<`              | 小于（数值）               | `["arg_rank", "<", 2]`                                      |
| `<=`             | 小于等于（数值）           | `["arg_rank", "<=", 2]`                                     |
| `v>`             | 大于（版本号）             | `["http_appv", ">", "1.2.3"]`                               |
| `v>=`            | 大于等于（版本号）         | `["http_appv", ">=", "1.2.3"]`                              |
| `v<`             | 小于（版本号）             | `["http_appv", "<", "1.2.3"]`                               |
| `v<=`            | 小于等于（版本号）         | `["http_appv", "<=", "1.2.3"]`                              |
| `ip~`，`ipmatch` | `ip` 匹配                  | `["remote_addr", "ip~", ["192.168.1.1", "192.168.2.0/24"]]` |
| `has`，`has*`    | 包含（左值为数组）         | `["custom_names", "has", "user1"]`                          |
| `in`，`in*`      | 存在（右值为数组）         | `["arg_name", "in", ["user1","user2"]]`                     |
| `rx~`，`rx~*`    | 同 `in` ，针对路径作了优化 | `["req_uri", "rx~", ["/path/to/1","/path/to/1"]]`           |


> 运算符中 `*` 表示忽略大小写



##### 1.1.2 逻辑运算符


| **运算符** | **描述** |
|---|---|
| `AND` | `AND(A,B)` 等价于 `A && B`. |
| `OR` | `OR(A,B)` 等价于 `A || B`. |
| `!AND` | `!AND(A,B)` 等价于 `!(A && B)`. |
| `!OR` | `!OR(A,B)` 等价于 `!(A || B)`. |


##### 1.1.3 变量

表达式中使用的变量，请参考 [内置变量说明](#9-内置变量说明)。


#### 1.2 配置事例

```json
        ["http_appv", "v>=", "1.2.3"],
        ["arg_name", "==", "user"],
        ["arg_test", "in", 
            ["T1", "T2"]
        ]
    ]
}
```


```json

    "demo_expr2": ["!AND",
        ["http_appv", "v>=", "1.2.3"],
        ["arg_name", "==", "user"],
        ["OR",
            ["arg_test", "==", "T1"],
            ["arg_test", "==", "T2"]
```
