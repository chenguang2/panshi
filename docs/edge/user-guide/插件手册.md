# EDGE 插件手册

本文档包含 EDGE v3.0 所有插件的分类特性汇总（优先级、Stream 支持、复用性等）以及各插件详细的配置参数说明、调用示例与注意事项。管理接口、安装配置等内容请参阅 [使用手册](使用手册.md)。

## 1. 插件的复用功能（别名调用）

为了支持在单个路由中多次执行同一个插件的不同逻辑，网关支持了**插件的复用功能**。

### 1.1 使用规则
- **可复用的插件**（未标注为 `unique` 的插件）可以在一个路由中通过别名的方式多次调用。
- **命名规范**：别名使用破折号 `-` 进行连接，格式为：`{原插件名}-{别名}`。每个主插件可以拥有多个不同的别名。
- **声明与注册**：别名插件依赖主插件的存在，且必须在 `edge.env` 的 `ex_plugins`（或 `ex_stream_plugins`）配置中进行声明注册。
- **独立属性**：所有声明的别名插件都可以像普通插件一样，在路由中配置自己独立的数据属性（Metadata Config / Config）。

### 1.2 注册示例
在部署配置 `edge.env` 中注册别名插件：
```yaml
ex_plugins:
  plugin_demo-test1: true      # 启用别名插件 1，使用默认优先级
  plugin_demo-test2: 1999      # 启用别名插件 2，并自定义其执行优先级为 1999
```

> [!WARNING]
> 凡是标注为 `unique` 的插件（即不可复用的插件，如 `cors`、`proxy_rewrite` 等），**不支持**使用别名进行多次实例化配置。

## auth_basic 插件使用文档

### 1. 插件概览

`auth_basic` 插件用于在网关层实现 HTTP 基础认证 (HTTP Basic Authentication)。它通过验证客户端在请求头中携带的 `Authorization` 信息，与网关中配置的**消费者 (Consumers)** 账号密码进行匹配，拦截未授权的访问。

主要应用场景：
- **服务访问受限防护**：保护敏感的后台管理页面、测试环境或仅限注册用户/合作伙伴调用的 API 接口。
- **简易多租户认证**：根据不同的客户端身份创建多个不同的 Consumer，实行差异化的访问授权与流量监控。

---

### 2. 配置详情

#### 2.1 插件静态属性 (Attr Config)

无。

---

#### 2.2 插件数据属性 (Metadata Config)

无。

---

#### 2.3 插件基础属性 (Config)

路由级配置。

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |
| `hide_credentials` | Boolean | `false` | 认证成功后，是否在将请求转发给上游前从 HTTP Header 中剥离删除 `Authorization` 认证标头。 | `true` |

---

#### 2.4 消费者属性 (Consumer Config)

在创建或更新消费者资源（API 路径：`/edge/admin/consumers`）时配置该用户的 Basic 认证凭证。

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |
| `username` | String | 无 | 必填。消费者 Basic 认证对应的用户名，建议与消费者的系统唯一 `username` 保持一致。 | `"test_user"` |
| `password` | String | 无 | 必填。消费者 Basic 认证对应的明文密码。 | `"super_secure_pass"` |

### 3. 调用与验证示例

要完整启用并验证 `auth_basic` 插件，共需配置两步：创建对应的消费者凭证，然后配置启用插件的路由。

> [!IMPORTANT]
> 调用示例中不包含真实敏感密钥，请将占位符 `API_KEY` 替换为您实际环境中的 Admin API Key。

#### 3.1 步骤 1：创建具有 Basic 凭证的消费者 (Consumer)

调用消费者接口，创建一个系统唯一名为 `company_a` 的消费者，并配置其 Basic 认证的用户名和密码。

```bash
curl -i -X PUT \
  -H "Content-Type: application/json" \
  -H "X-Api-Key: API_KEY" \
  -d '{
    "username": "company_a",
    "name": "Company A Consumer",
    "desc": "Consumer for B2B Client A",
    "plugins": {
      "auth_basic": {
        "username": "client_a",
        "password": "CLIENT_A_PASSWORD"
      }
    }
  }' \
  http://127.0.0.1:9991/edge/admin/consumers
```

#### 3.2 步骤 2：创建启用 auth_basic 插件的路由

创建路由关联路径 `/api/secure`，并开启 `auth_basic` 插件，配置为隐藏敏感认证信息（`hide_credentials: true`），避免凭证透传至后端。

```bash
curl -i -X PUT \
  -H "Content-Type: application/json" \
  -H "X-Api-Key: API_KEY" \
  -d '{
    "uri": "/api/secure",
    "plugins": {
      "auth_basic": {
        "hide_credentials": true
      }
    },
    "upstream": {
      "nodes": {
        "127.0.0.1:8811": 1
      },
      "type": "roundrobin"
    }
  }' \
  http://127.0.0.1:9991/edge/admin/routes/1004
```

---

#### 3.3 访问网关进行验证

##### 1. 未携带认证信息的匿名访问（预期被拦截）

向网关发起匿名请求：

```bash
curl -i http://127.0.0.1:9981/api/secure
```

**预期响应输出 (返回 401 并附带 WWW-Authenticate 头)**：

```http
HTTP/1.1 401 Unauthorized
Connection: keep-alive
WWW-Authenticate: Basic realm='.'
Content-Length: 42
Content-Type: application/json

{"message":"Missing authorization in request"}
```

##### 2. 使用错误的凭证访问（预期被拒绝）

模拟输入错误的密码：

```bash
curl -i -u "client_a:WRONG_PASSWORD" http://127.0.0.1:9981/api/secure
```

**预期响应输出**：

```http
HTTP/1.1 401 Unauthorized
Connection: keep-alive
WWW-Authenticate: Basic realm='.'
Content-Length: 35
Content-Type: application/json

{"message":"Invalid user authorization"}
```

##### 3. 使用正确的凭证访问（预期通过）

使用创建消费者时配置的正确用户名和密码发起请求：

```bash
curl -i -u "client_a:CLIENT_A_PASSWORD" http://127.0.0.1:9981/api/secure
```

**预期响应输出**：

```http
HTTP/1.1 200 OK
Content-Type: text/plain
Connection: keep-alive
...
```
*(注意：由于设置了 `hide_credentials: true`，后端上游服务器收到该请求时，其 Header 中将不包含 `Authorization` 头)*

## auth_key 插件使用文档

### 1. 插件概览

`auth_key` 插件通过识别请求中指定的 API Key (密钥) 进行身份验证。它支持从指定的请求头 (Header) 或 URL 查询参数 (Query Parameter) 中获取密钥，并与绑定的**消费者 (Consumers)** 进行匹配，起到接口调用鉴权及访问审计的作用。

主要应用场景：
- **开放 API 服务接入**：对外提供数据订阅或计算接口时，为合作方发放唯一的 API Key，用以控制访问权限和监控流量消耗。

## breaker 插件使用文档

### 1. 插件概览

`breaker` 插件用于在 API 网关接收到请求时提供服务熔断保护（Circuit Breaker）。当检测到后端服务的特定异常（如 `5xx` 响应）达到设定的阈值时，网关会开启熔断保护，在指定的保护期内自动拒绝后续的超频流量，并直接向客户端返回友好的降级响应（如 `503`），避免因个别后端故障拖垮整个网关或服务链条。

主要应用场景：
- **微服务高可用防护**：防止某个微服务节点的宕机或超时导致线程积压，进而引发系统雪崩。
- **降级应急控制**：对已知的非核心业务进行异常阈值限制，当故障超出阈值时实现自动化切流与快速降级。

---

> **[!IMPORTANT]**
> **所有插件的数据属性（Metadata Config）变更后，必须执行一次插件热加载操作以使配置生效：**
> - **请求方法**：`PUT`
> - **请求路径**：`/edge/admin/plugins/reload`
> 
> *您可以在配置多个插件的数据属性后，统一执行一次热加载操作。*

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |
| `status` | Integer | `503` | 触发熔断被拦截时，网关返回给客户端的 HTTP 状态码。取值范围 `200` 至 `599`。 | `503` |
| `message` | String | `"Your request is not allowed."` | 触发熔断被拦截时，返回的 HTTP 响应体消息内容（最长 1024 字符）。 | `"Service temporarily unavailable."` |
| `bypass_missing_key` | Boolean | `true` | 当 `key` 解析失败或变量缺失返回 `nil` 时，是否跳过熔断逻辑。如果设为 `false`，当 Key 缺失时将直接触发熔断拦截。 | `true` |
| `bypass_error_limit` | Boolean | `true` | 当本地限流共享内存发生溢出或异常时，是否跳过限制（放行流量）。如果设为 `false`，遇到内部故障将直接触发熔断拦截。 | `true` |
| `header_name` | String | `"X-Edge-Breaker-By"` | 拦截响应时，若开启了展示，该字段定义响应标头（Header）的名称。 | `"X-Circuit-Status"` |
| `show_breaker_header` | Boolean | `false` | 拦截响应中是否添加定义在 `header_name` 里的响应头，以标识该响应来自网关的熔断拦截。 | `true` |

#### 2.3 插件基础属性 (Config)

路由级配置，可覆盖全局定义的插件数据属性 (Metadata Config)。

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |
| `key` | String | `"${uri}"` | 熔断器进行跟踪及流量限制的 Key。支持变量插值（例如通过 `${uri}`、`${host}` 等）。 | `"${uri}"` |
| `time` | Integer | `3` | 最大熔断持续保护时间（秒）。每次触发异常会以指数退避算法计算持续时间（即 $2^{\text{failure\_times}}$ 秒），但最大不超过该时间。 | `10` |
| `count` | Integer | `3` | 处于熔断保护期内时，允许放行测试的最大请求流量个数。 | `5` |
| `window` | Number | `1` | 配合 `count` 的流量限制统计时间窗口（秒）。 | `2` |
| `unhealthy` | Object | 无 | 异常监控状态设置。包含具体的匹配规则与触发阈值，详见下方**说明**。 | 见配置示例 |

| `status` | Integer | `503` | 同上 | `503` |
| `message` | String | `"Your request is not allowed."` | 同上 | `"Service Unavailable"` |
| `bypass_missing_key` | Boolean | `true` | 同上 | `true` |
| `bypass_error_limit` | Boolean | `true` | 同上 | `true` |
| `header_name` | String | `"X-Edge-Breaker-By"` | 同上 | `"X-Breaker-Status"` |
| `show_breaker_header` | Boolean | `false` | 同上 | `true` |

> **说明**
> 
> *   **`unhealthy` 的子字段说明**：
>     *   `expr`：标准 Edge 条件表达式列表，用于确定哪些上游响应是不健康的。默认值为 `[ ["ngx_ups_status", ">=", 500] ]`。关于条件表达式的写法细节，请参考 [Edge 表达式列表](使用手册.md#8-核心表达式引擎)。
>     *   `count`：触发熔断所需的连续不健康响应次数，默认值为 `3`。
> 
> *   **`healthy` 的子字段说明**：
>     *   `expr`：标准 Edge 条件表达式列表，用于确定哪些上游响应是健康的。默认值为 `[ ["ngx_ups_status", "==", 200] ]`。详见 [Edge 表达式列表](使用手册.md#8-核心表达式引擎)。
>     *   `count`：重置熔断器所需的连续健康响应次数，默认值为 `1`。

使用 Admin 管理接口创建或更新路由，为路径 `/api/breaker` 开启 `breaker` 插件。
我们配置熔断器跟踪 Key 为当前请求的 URI：
- 异常条件为：连续 `3` 次返回状态码 $\ge 500$（`unhealthy` 规则）。
- 触发熔断后的最长保护时长为 `10` 秒。
- 熔断期间允许每秒放行 `1` 次请求用于测试后端恢复状况（`count: 1`, `window: 1`）。
- 恢复正常条件为：放行测试请求得到连续 `2` 次 `200` 正常响应（`healthy` 规则）。
- 被拦截时返回 `503` 及 `"Service Broken"`，并在响应头中添加 `X-Breaker-Blocked: true` 的标识。

> [!IMPORTANT]
> 调用示例中不包含真实敏感密钥，请将占位符 `API_KEY` 替换为您实际环境中的 Admin API Key。

```bash

  -H "Content-Type: application/json" \
  -H "X-Api-Key: API_KEY" \
  -d '{
    "uri": "/api/breaker",
    "plugins": {
      "breaker": {
        "key": "${uri}",
        "time": 10,
        "count": 1,
        "window": 1,
        "unhealthy": {
          "expr": [ ["ngx_ups_status", ">=", 500] ],
          "count": 3
        },
        "healthy": {
          "expr": [ ["ngx_ups_status", "==", 200] ],
          "count": 2
        },
        "status": 503,
        "message": "Service Broken",
        "header_name": "X-Breaker-Blocked",
        "show_breaker_header": true
      }
    },
    "upstream": {
      "nodes": {
        "127.0.0.1:8811": 1
      },
      "type": "roundrobin"
    }
  }' \
  http://127.0.0.1:9991/edge/admin/routes/1006
```

#### 3.2 访问网关进行验证

##### 1. 模拟后端异常触发熔断

向网关发起 `3` 次能够使后端返回 `500` 的异常请求（假设 `/api/breaker` 接口对应的后端支持通过查询参数 `error=true` 故意返回 `500`）：

```bash

curl -i http://127.0.0.1:9981/api/breaker?error=true
curl -i http://127.0.0.1:9981/api/breaker?error=true
curl -i http://127.0.0.1:9981/api/breaker?error=true
```

**第 3 次请求返回后，熔断器被激活开启。**

##### 2. 模拟高频访问触发网关直接拦截

在激活状态下，向网关以高频方式发起普通正常请求。因为我们在熔断期只允许每秒放行 `1` 次（`count: 1`, `window: 1`），超出限额的请求会被网关直接拒绝：

```bash
## 1 秒内连续发送 2 次普通请求

curl -i http://127.0.0.1:9981/api/breaker
```

**预期第 2 次超额请求的响应输出（网关直接返回 503 且拦截头生效）**：

```http

Content-Type: text/plain
Connection: keep-alive
X-Breaker-Blocked: EDGE/3.0
Content-Length: 14

Service Broken
```

##### 3. 模拟后端服务恢复，重置熔断器

停止高频请求，让测试请求成功投递到后端并得到 `200 OK` 正常响应（每秒发送 1 次以确保不触发放行限额拦截，且不带 `error=true` 报错参数）：

```bash
## 每秒发送 1 次请求，使其被半开测试放行到后端，连续 2 次

curl -i http://127.0.0.1:9981/api/breaker
sleep 1
curl -i http://127.0.0.1:9981/api/breaker
```

**当前端连续 2 次得到健康响应后，熔断器自动切换为关闭（Closed）状态。**

此后再次高频请求，网关不再进行限速拦截：

```bash

curl -i http://127.0.0.1:9981/api/breaker
```

## cors 插件使用文档

### 1. 插件概览

`cors` 插件用于为 HTTP 服务自动注入跨域资源共享 (CORS) 相关的响应标头（如 `Access-Control-Allow-Origin` 等），并自动拦截处理浏览器的 `OPTIONS` 预检请求，使前端应用可以安全地跨域调用后端的 API 接口。

主要应用场景：
- **跨域接口调用**：前后端分离架构中，前端网页（如 `https://web.example.com`）需要请求网关下的 API 接口（如 `https://api.example.com`）。
- **多域名跨域共享**：网关支持使用正则表达式或全局元数据定义受信任的域名范围，为这些特定域名提供动态跨域响应。

---

### 2. 配置详情

#### 2.1 插件静态属性 (Attr Config)

无。

---

#### 2.2 插件数据属性 (Metadata Config)

通过管理接口进行全局配置（API 路径：`/edge/admin/plugin_metadata/cors`），用于设定全局 Origin 分组映射表。

> **[!IMPORTANT]**
> **所有插件的数据属性（Metadata Config）变更后，必须执行一次插件热加载操作以使配置生效：**
> - **请求方法**：`PUT`
> - **请求路径**：`/edge/admin/plugins/reload`
> 
> *您可以在配置多个插件的数据属性后，统一执行一次热加载操作。*

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |

---

#### 2.3 插件基础属性 (Config)

路由级配置，可覆盖全局定义的插件数据属性 (Metadata Config)。

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |
| `allow_origins` | String | `"*"` | 允许跨域访问的 Origin 源。支持具体的 Origin 地址、以逗号分隔的多个 Origin 地址、`*`（允许所有）或 `**`（动态镜像请求中的 Origin 头）。格式必须匹配 `^(\*\|\*\*\|null\|\w+://[^,]+(,\w+://[^,]+)*)$`。 | `"https://example.com"` |
| `allow_methods` | String | `"*"` | 允许的跨域 HTTP 方法。支持 `*`、以逗号分隔的多个方法（如 `GET,POST`）或 `**`（转换为全部标准方法：`GET,POST,PUT,DELETE,PATCH,HEAD,OPTIONS,CONNECT,TRACE`）。 | `"GET,POST,OPTIONS"` |
| `allow_headers` | String | `"*"` | 允许的跨域请求标头。支持逗号分隔的具体标头、`*` 或 `**`（动态获取请求头 `Access-Control-Request-Headers` 的值）。 | `"Content-Type,Authorization"` |
| `expose_headers` | String | 无 | 允许浏览器访问的响应标头列表。多个值使用逗号分隔。 | `"X-Custom-Header,X-Trace-Id"` |
| `max_age` | Integer | `5` | 预检请求（OPTIONS）结果的浏览器缓存时间，单位为秒。 | `3600` |
| `allow_credential` | Boolean | `false` | 是否允许浏览器在跨域请求中携带凭证（如 Cookie, Authorization 标头等）。 | `true` |
| `allow_origins_by_regex` | Array | 无 | 用于正则匹配允许的 Origin 列表。数组成员为正则表达式。 | `["^https://.*\\.domain\\.com$"]` |
| `allow_origins_by_metadata` | Array | 无 | 引用全局数据属性 (Metadata Config) 中定义的 `allow_origins` 的键名（分组名）列表。若匹配成功，则将该 Origin 视为合法。 | `["group1"]` |
| `timing_allow_origins` | String | 无 | 允许设置 `Timing-Allow-Origin` 响应头的 Origin 规则，用于资源定时 API 收集。格式规范与 `allow_origins` 一致。 | `"https://example.com"` |

---

#### 2.4 注意事项

*   **凭证安全限制**：当开启 `allow_credential: true` 时，**严禁**将 `allow_origins`、`allow_methods`、`allow_headers`、`expose_headers`、`timing_allow_origins` 设为 `*`，否则校验将报错拒绝。
*   **Vary 标头**：当配置具体的跨域 Origin（非 `*`）时，网关将自动在响应头中加入 `Vary: Origin`，防止不同域的浏览器读取到代理缓存中前一个域的跨域配置响应。

## log_mirror 插件使用文档

### 1. 插件概览

`log_mirror` 插件允许网关在不影响主客户端请求流程的前提下，异步克隆当前真实的 HTTP 请求（包括请求方法、URI、查询参数、请求体及标头），并向指定的第三方接收端发起镜像请求，常用于流量回放、旁路 WAF 审计或灰度比对。

主要应用场景：
- **生产流量克隆与测试**：将真实的在线业务流量镜像复制到灰度环境，进行影子系统压力测试或新版功能比对。
- **安全旁路检测**：在后台无感复制一份完整的 HTTP 请求报文并投递至安全审计中心。

---

### 2. 配置详情

#### 2.1 插件静态属性 (Attr Config)

无。

---

#### 2.2 插件数据属性 (Metadata Config)

无。

---

#### 2.3 插件基础属性 (Config)

路由级配置。

| `logs` | String / Array | 无 | 必填。指定要推送到的一个或多个镜像接收端 URI。 | `"http://127.0.0.1:9980/mirror/test"` |

## log_process 使用文档

### 插件概览

`log_process` 插件是网关提供的高性能文件日志收集与记录插件。支持对客户端请求和响应的多种内置变量进行插值拼接，并生成日志文件落地磁盘。支持缓存批量写盘（提高吞吐性能）以及针对特定请求/响应体（通过条件表达式）的过滤提取与记录。

---

### 配置详情

#### 1. 插件静态属性 (Attr Config)

无。

---

#### 2. 插件数据属性 (Metadata Config)

通过管理节点接口进行全局配置（路径：`/edge/admin/plugin_metadata/log_process`）。**配置变更后，必须执行一次插件热加载操作以使配置生效（`PUT /edge/admin/plugins/reload`，详见 [插件管理](使用手册.md#插件管理-plugins)）；如果同时配置了多个插件的数据属性，可以在全部配置完成后统一执行一次热加载**。

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |
| `logs` | Object/Array | 见下文说明 | 默认的文件日志配置。可以是以日志文件路径为键、值为单项配置的字典，或者是单项配置组成的数组。 | 见下文示例 |
| `formats_sep` | String | `"\|;"` | 默认的日志字段分隔符。 | `"\|"` |
| `json` | String | 无 | 处理json数据时，dkjson会按字段顺序进行排序，cjson不排序； | `"cjson"` |
| `cache_time` | Integer | `1` | 目前，只支持0，1 | `1` |

> **说明**: `logs` 内每个日志项包含以下子字段：
> - `file`：日志目标路径。仅支持字母、数字、点、正斜杠、下划线及减号（例如：`logs/process.log`）。
> - `formats`：日志字段格式，支持字符串、数组或包含自定义键值映射的对象。如果是数组，将使用 `formats_sep` 作为分隔符拼接输出。
> - `formats_sep`：指定覆盖全局的日志字段分隔符。
> - `json`：指定覆盖全局的 JSON 库（`"cjson"` 或 `"dkjson"`）。
> - `cache_time`：指定覆盖全局的日志缓存落盘时间。

---

#### 3. 插件基础属性 (Config)

路由级属性，启用该插件时配置在路由下。

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |
| `logs` | String/Array | `"logs/process.log"` (HTTP)<br>`"logs/process.stream.log"` (Stream) | 指定记录日志的磁盘文件路径，支持配置单个路径字符串或由多个路径组成的数组。详见下方注意事项。 | `["logs/custom_access.log"]` |
| `formats` | String/Object/Array | 见全局默认格式 | 覆盖全局的日志字段格式。可配置变量详见下方说明。 | `["${time_local}", "${remote_addr}", "${status}"]` |
| `formats_sep` | String | 同上 | 覆盖默认的日志字段分隔符。 | `"\|"` |
| `json` | String | 同上 | 处理json数据时，dkjson会按字段顺序进行排序，cjson不排序； | `"dkjson"` |
| `cache_time` | Integer | 同上 | 目前，只支持0，1 | `0` |
| `include_req_body` | Boolean | `false` | 处理请求时，即使没有使用到请求体，也要为稍后的日志，预先保存好请求体 | `true` |
| `include_req_body_expr` | Array | 无 | 条件表达式。只有在此条件匹配成功时，才会记录请求体。详见下方说明。 | `[ ["uri", "==", "/submit"] ]` |
| `include_resp_body` | Boolean | `false` | 处理请求时，即使没有使用到响应体，也要为稍后的日志，预先保存好响应体 | `true` |
| `include_resp_body_expr` | Array | 无 | 条件表达式。只有在此条件匹配成功时，才会记录响应体。详见下方说明。 | `[ ["status", ">=", 400] ]` |

> **说明**:
> 1. `formats` 支持配置 Nginx/Edge 变量进行动态提取，可用的变量参考 [Edge 变量列表](使用手册.md#9-内置变量说明)。
> 2. 条件表达式（如 `include_req_body_expr`、`include_resp_body_expr` 等）的示例及配置必须遵循标准的 Edge 条件表达式格式，详见 [Edge 表达式列表](使用手册.md#8-核心表达式引擎)。
> 3. **全局默认缺省目的地与日志格式**：
>     当提交的插件数据属性 (Metadata Config) 为 `{}` 或 `{ "logs": {} }` 时，网关会自动配置默认日志路径与默认格式：
>     *   **HTTP 模式**：
>         *   **默认日志文件路径**：`logs/process.log`
>         *   **默认日志格式**：如果不指定 `formats`，网关将使用以下内置变量列表依次拼接输出（默认使用 `|;` 分隔）：
>             1. `${req_start_time#time_format,%Y%m%d%H%M%S,%03d}`（请求开始时间，精确到毫秒）
>             2. `${http_x-edge-traceid}`（Trace ID）
>             3. `${clientip}`（真实客户端 IP）
>             4. `${username}`（经过认证的用户标识）
>             5. `${deviceid}`（移动客户端设备 ID）
>             6. `${udid}`（移动设备唯一标识码）
>             7. `${cookie_JSESSIONID}`（客户端 Cookie 中的 JSESSIONID）
>             8. `${method}`（HTTP 请求方法）
>             9. `${uri}`（请求 URI）
>             10. `${req_args_string#limitn,2048}`（请求查询参数，限长 2048 字节）
>             11. `${http_referer}`（HTTP 请求中的 Referer 来源）
>             12. `${http_cookie}`（完整的客户端 Cookie 串）
>             13. `${http_user-agent}`（客户端 User-Agent）
>             14. `${req_headers}`（完整的请求头部信息）
>             15. `${upstream_addr}`（转发的上游后端服务器地址）
>             16. `${upstream_response_time#fixdefault,0,0}`（上游后端响应耗时，缺失时缺省值为 `0`）
>             17. `${request_time}`（网关处理请求的总耗时）
>             18. `${status}`（网关返回给客户端的 HTTP 状态码）
>             19. `${route_id}`（匹配命中的网关路由 ID）
>             20. `${plugin_riskinfo}`（风控等插件评估出的风险特征描述信息）
>             21. `${plugin_riskid#fixdefault,,0}`（安全风控策略 ID，缺失时缺省值为 `0`）
>     *   **Stream 模式**：
>         *   **默认日志文件路径**：`logs/process.stream.log`
>         *   **默认日志格式**：如果不指定 `formats`，网关将使用以下内置变量依次拼接（默认使用 `|;` 分隔）：
>             1. `${host}`（请求的主机名）
>             2. `${time_iso8601}`（ISO 8601 格式的本地时间）
>             3. `${remote_addr}`（客户端 IP 地址）

---

#### 注意事项

- **动态变量解析**：在 `formats` 配置中写入的所有动态变量，如 `${uri}`、`${host}` 等，都将被网关动态提取解析。支持的内置变量请参考 [Edge 变量列表](使用手册.md#9-内置变量说明)。
- **日志格式与文件配置**：日志属性（特别是日志格式如 `formats`）应优先配置在**数据属性 (Metadata Config)** 中，而在**基础属性 (Config)** 里建议仅配置需要写入的日志文件路径。这可保证所有写入同一文件的日志格式保持统一。如果确实需要在路由的基础属性中定义格式，应当格外注意多路由写入同一文件时格式是否一致；此外，当指定的日志文件第一次被调用且在数据属性中尚未初始化配置时，错误日志中可能会出现类似 `not found log_conf: logs/xxx.log` 的警告。
- **磁盘写入延迟**：为了最大化网关的处理吞吐率，日志支持内存批量落盘。如果设置了 `cache_time` 大于 `0`，写盘操作会有最长 `cache_time` 秒的延迟。若需进行实时调试，可将 `cache_time` 设置为 `0`（但高并发生产环境不推荐设置 `0`）。
- **表达式过滤 Body**：保存请求体与响应体会占用额外的内存，并且响应体缓冲会改变 Content-Length 的发送方式。推荐仅针对必要的调试接口或异常状态码通过 `include_req_body_expr` 与 `include_resp_body_expr` 进行条件性开启。

## log_rotate 使用文档

### 插件概览

`log_rotate` 插件是网关提供的高性能本地日志切分、打包压缩与生命周期清理插件。该插件作为全局单例运行，无需在特定路由或 Consumer 规则下挂载。网关一旦初始化并启动，该插件即会在后台以定时器（Timer）驱动，自动针对指定的日志路径进行周期性的重命名切割、按需进行 gzip 压缩归档以及超出最大保存保留数量后的超期文件清理。

---

### 配置详情

#### 1. 插件静态属性 (Attr Config)

该属性配置存储于 `edge.env` 文件的 `plugin_attr.log_rotate` 配置块中。

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |
| `enable` | Boolean | `true` | 全局开启或关闭日志轮转功能。 | `true` |
| `logs` | Object | 见下文说明 | 包含每个日志文件的具体轮转规则。以日志文件相对路径为键。若未配置，默认只轮转系统自带日志。 | 见下文示例 |
| `rotate` | Integer | `48` | 全局默认的轮转历史日志最多保留数量。设置为负值表示不限制保留数量。 | `24` |
| `interval` | String | `"1h"` | 全局默认的日志轮转周期。支持按月、按日或按小时轮转。 | `"1d"` |
| `compress` | Boolean | `true` | 全局默认的是否将轮转切割出的文件打包压缩为 `.tar.gz`。 | `true` |
| `compress_delay` | Boolean | `false` | 全局默认的是否延迟压缩。若为 `true`，则会将当前切分出来的日志延迟到下一个轮转周期再进行压缩，平抑瞬间耗能。 | `true` |

##### **`logs` 字典内单项配置字段说明**：

`logs` 中的每一个键名（例如 `logs/process.log`）必须是匹配正则 `^[a-zA-Z0-9./_-]+$` 的合法相对路径。单项配置对象所包含的子字段如下：
- `enable` (Boolean)：该日志文件特有的是否开启轮转的配置。
- `rotate` (Integer)：该日志文件特有的保留份数。若配置，则覆盖全局配置。
- `interval` (String)：该日志文件特有的轮转周期。可选值有：`"1mon"`, `"1d"`, `"1h"`。
- `compress` (Boolean)：该日志文件特有的是否压缩选项。
- `compress_delay` (Boolean)：该日志文件特有的是否延迟压缩选项。

> **说明**：
> 1. 如果 `logs` 未进行任何显式配置，系统只会为 `logs/error.log` 提供默认的轮转（默认轮转周期为 `1d`，保留份数为 `3`），而 `logs/access.log` 默认不轮转。
> 2. `log_rotate` 配置继承 fallback 链：**单项日志具体配置** -> **全局插件配置 (plugin_attr.log_rotate)** -> **系统默认缺省值**。

无。

#### 3. 插件基础属性 (Config)

无。

#### 注意事项

- **配置应用流程**：由于 `log_rotate` 为全局静态属性配置，通过修改 `edge.env` 完成后，需要通过命令行重新初始化与重载才能被后台定时任务读取生效。
- **重命名格式**：系统在轮转切割时，会在原文件名与扩展名之间嵌入当前计算出来的时间后缀。例如：
  - 按小时轮转 (`"1h"`)：`process.log` -> `process.2026061404.log`
  - 按天轮转 (`"1d"`)：`process.log` -> `process.20260614.log`
  - 按月轮转 (`"1mon"`)：`process.log` -> `process.202606.log`
- **防 CPU 峰值开销**：当管理的日志文件较大时，打包压缩（`compress`）会在瞬间带来较高的 CPU 负载。建议将 `compress_delay` 设置为 `true`。它会采用延迟压缩机制，并将压缩操作放入下一次轮转周期后，且配合随机抖动时间执行，有效降低瞬时的 I/O 与 CPU 开销。

---

### 启用与配置示例

#### 1. 修改 `edge.env` 配置属性

在 `edge.env` 文件的 `plugin_attr` 中，按需添加或修改 `log_rotate` 部分：

```yaml
plugin_attr:
  log_rotate:
    interval: 1h               # 全局默认的日志轮转周期：每小时轮转一次
    rotate: 48                 # 全局默认保留的最新的轮转日志备份份数
    compress: true             # 全局默认对切分出来的日志进行打包压缩
    compress_delay: true       # 全局默认开启延迟压缩，缓解 CPU 负载压力
    logs:
      logs/access.log:
        enable: false
      logs/error.log:
        interval: 1d
        rotate: 3
      logs/process.log:        # 对自定义的 log_process 插件产生的日志执行轮转配置
        interval: 1d
        rotate: 30
```

```bash
## 重载网关服务以使静态属性和后台定时器生效
```

## log_syslog 插件使用文档

### 1. 插件概览

`log_syslog` 插件用于收集网关层请求与响应的访问日志，并将其通过 TCP、UDP 或 TLS 协议异步批量推送到远程的 Syslog 日志中心。这有利于企业实施集中式安全审计、流量监控以及问题排查。

主要应用场景：
- **集中式日志分析**：向 Splunk、Logstash 或 Graylog 等日志中心推送标准的访问日志。
- **特定异常流量审计**：配合条件过滤表达式，只将请求体或响应体中异常的数据（例如 5xx 响应）记录发送给 Syslog，用于故障诊断。

> *   **默认日志格式 (HTTP 模式)**：

>     1. `req_start_time` (请求开始时间，格式化为 `%Y%m%d%H%M%S.%03d`)
>     2. `http_x-edge-traceid` (Trace ID)
>     3. `clientip` (客户端 IP)
>     4. `username` (登录名)
>     5. `deviceid` (设备 ID)
>     6. `udid` (UDID)
>     7. `cookie_JSESSIONID` (JSESSIONID cookie 值)
>     8. `method` (HTTP 方法)
>     9. `uri` (请求 URI)
>     10. `req_args_string` (请求 Query 参数，最长截取 2048 字节)
>     11. `http_referer` (Referer 头)
>     12. `http_cookie` (Cookie 头)
>     13. `http_user-agent` (User-Agent 头)
>     14. `req_headers` (完整请求头)
>     15. `upstream_addr` (上游转发目标地址)
>     16. `upstream_response_time` (上游响应时间)
>     17. `request_time` (网关总处理耗时)
>     18. `status` (响应状态码)
>     19. `route_id` (匹配的路由 ID)
>     20. `plugin_riskinfo` (安全风险信息)
>     21. `plugin_riskid` (触发拦截的安全规则 ID)
> 
> *   **默认日志格式 (Stream 模式)**：
>     在非 HTTP (Stream TCP/UDP) 代理模式下，默认采用以下格式进行日志传输：
>     1. `host` (网关主机名)
>     2. `time_iso8601` (ISO8601 格式时间戳)
>     3. `remote_addr` (客户端连接源 IP)

### 3. 调用与验证示例

#### 3.1 创建并启用路由

使用 Admin 管理接口创建或更新路由，为路径 `/api/log` 开启 `log_syslog` 插件。
- 日志输出目标为本地部署的 Syslog 服务：`tcp://127.0.0.1:514`。
- 自定义输出格式中仅记录：请求时间、客户端 IP、URI、响应状态码、以及 Trace ID。
- 开启响应体备份，并且只记录响应状态码大于等于 `500` 时的响应体数据（`include_resp_body_expr` 规则）。

> [!IMPORTANT]
> 调用示例中不包含真实敏感密钥，请将占位符 `API_KEY` 替换为您实际环境中的 Admin API Key。

```bash
curl -i -X PUT \
  -H "Content-Type: application/json" \
  -H "X-Api-Key: API_KEY" \
  -d '{
    "uri": "/api/log",
    "plugins": {
      "log_syslog": {
        "logs": ["tcp://127.0.0.1:514"],
        "formats": {
          "time": "${req_start_time#time_format,%Y%m%d%H%M%S,%03d}",
          "client": "${clientip}",
          "uri": "${uri}",
          "status": "${status}",
          "trace_id": "${http_x-edge-traceid}"
        },
        "include_resp_body": true,
        "include_resp_body_expr": [ ["status", ">=", 500] ]
      }
    },
    "upstream": {
      "nodes": {
        "127.0.0.1:8811": 1
      },
      "type": "roundrobin"
    }
  }' \
  http://127.0.0.1:9991/edge/admin/routes/1007
```

#### 3.2 访问网关进行验证

##### 发送普通请求

向网关发起一次普通访问请求：

```bash

```

**预期 Syslog 服务端接收到的日志流输出 (单行 JSON)**：

```json

```

##### 发送错误请求触发响应体记录

向网关发起能够引发上游返回 `500` 的异常请求：


**预期 Syslog 服务端接收到的日志流输出**：
日志输出的 JSON 消息体中，会自动包含上游返回的响应体明文信息：

若网关无法连通配置的 Syslog 服务器（如 `127.0.0.1:514` 端口未开启），由于启用了批处理失败保护，网关会将日志降级写入本地：

```bash

```

**预期本地文件输出**：
文件中将包含对应的日志字符串明文，以防止日志丢失。

## mocking 使用文档

### 插件概览

`mocking` 插件用于在 Edge 网关层拦截请求并直接返回预定义的模拟响应。它支持返回固定内容，也支持根据 JSON Schema 动态生成随机响应，并允许自定义状态码、响应头及响应延迟。

### 配置详情

#### 插件静态属性 (Attr Config)

无。

#### 插件数据属性 (Metadata Config)

全局默认配置。
- **接口路径**: `/edge/admin/plugin_metadata/mocking`
- **属性列表**:

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |
| **delay** | Number | `0` | 响应延迟时间（秒）。 | `0.5` |
| **status** | Integer | `200` | 拦截响应的状态码 (200-599)。 | `201` |
| **content** | String/Object | 无 | 固定响应内容。支持字符串或 JSON 对象。支持变量。 | `"{\"msg\": \"success\"}"` |
| **content_schema** | Object | 无 | JSON Schema 定义。若配置则根据 Schema 生成随机数据。 | `{"type": "object", "properties": {"id": {"type": "integer"}}}` |
| **content_type** | String | `application/json;charset=utf8` | 响应的 Content-Type。 | `"application/xml"` |
| **headers** | Object | 无 | 自定义响应头，支持变量。 | `{"X-Server": "Edge"}` |
| **header_name** | String | `"X-Mock-By"` | Mock 标识头的名称。 | `"X-Edge-Mock"` |
| **show_mock_header**| Boolean | `false` | 是否在响应中包含 Mock 标识头。 | `true` |
| **xml_root_tag** | String | `"data"` | 当 Content-Type 为 XML 且使用 Schema 时，XML 的根节点名称。 | `"root"` |

- **设置示例**:
```bash
curl -L -X PUT 'http://[ADMIN_ADDR]/edge/admin/plugin_metadata/mocking' \
-H 'X-API-KEY: api-key' \
-H 'Content-Type: application/json' \
-d '{
    "header_name": "X-Edge-Mock",
    "show_mock_header": true,
    "delay": 0.1
}'
```

#### 插件基础属性 (Config)

同上。

#### 注意事项

1.  **优先级**: `content` 的优先级高于 `content_schema`。若两者同时存在，优先使用 `content`。
2.  **Schema 支持**: `content_schema` 自动生成功能目前仅在 `content_type` 为 `application/json` 或 `application/xml`（及其变体）时生效。
3.  **路由覆盖**: 路由级的 `Config` 属性将完全覆盖 `Metadata` 中对应的同名字段。
4.  **变量支持**: 插件支持在 `content` 及 `headers` 中使用动态变量。变量列表请参考 [Edge 变量 (包含 Nginx 变量)](使用手册.md#9-内置变量说明)。

| `prefer_route_name` | Boolean | `false` | 是否使用路由或服务的名称（Name）来标识命中的路由或服务。若为 `false`，则使用内部 ID 标识。 | `true` |
| `prefer_restrict_uri` | Boolean | `true` | 监控中记录的匹配 URI 是否优先展示为 URI白名单插件允许的 URI 地址，设置为 false 则仅使用路由配置的 URI 地址。 | `true` |
| `plugin_restrict_uri` | String \| Array | `"^security_restrict_uri"` | 当 `prefer_restrict_uri` 开启时，触发拦截匹配的插件正则模式或插件名称列表。 | `"^security_restrict_uri"` |
| `custom_uri` | String | 无 | 用于统计的自定义 URI，支持网关内置动态变量（如 `${uri}` 等）。 | `"${uri}"` |

---

#### 注意事项

- **命名规范**：如果在路由级基础配置中将 `prefer_route_name` 设为 `true`，建议对网关中所有的路由和服务进行统一、规范的英文或拼音命名。这是因为路由和服务名称在系统中没有强唯一性限制，不规范的命名可能会导致指标合并后难以区分。
- **变量支持**：`custom_uri` 字段支持使用网关内置变量，常用的变量如 `${uri}`（请求 URI 路径）、`${host}`（请求主机名）等。完整变量支持详情请引用：[Edge 变量列表](使用手册.md#9-内置变量说明)。

### 监控指标列表

启用 `monitor` 插件后，网关将向 Prometheus 导出并提供以下关键度量数据指标以供查看和绘图：

#### HTTP 阶段监控指标
- `edge_bandwidth`：服务消耗的物理总流量大小（单位：字节）。可以通过 `type="ingress"`（入流量）和 `type="egress"`（出流量）区分流量方向。
- `edge_http_requests_total`：自网关启动以来累计处理的客户端 HTTP 请求总量。
- `edge_http_status`：响应状态码统计，可通过 `code` 标签（如 `200`、`404`、`502` 等）对各路由和 matched_uri 的命中频次进行详细观测。
- `edge_http_latency`：HTTP 请求各类耗时的分布直方图（单位：毫秒）。可通过 `type` 区分总耗时（`request`）、后端连接耗时（`upstream`）及网关自身开销（`edge`）等。导出的直方图指标包含以下三项：
  - `edge_http_latency_bucket`：耗时计数，统计 `10, 20, 50, 100, 200, 500, 1000, 2000, 5000, 10000ms` 以及大于 10000ms 的请求耗时分布数。
  - `edge_http_latency_count`：各耗时来源的请求计数。
  - `edge_http_latency_sum`：各耗时来源的累计总请求耗时。
- `edge_nginx_http_current_connections`：网关底层 Nginx 的当前各种连接状态数据统计（如活跃连接、等待连接等）。
- `edge_shared_dict_capacity_bytes` / `edge_shared_dict_free_space_bytes`：网关内部各个共享内存字典的总体容量与剩余空闲空间监控。
- `edge_upstream_status`：各个后端上游节点健康检查状态的存活率指标。
- `edge_plugin_risk` / `edge_plugin_riskid`：网关各防守安全插件的累计拦截统计，支持下钻细化到具体的 WAF 规则 ID。
- `edge_plugin_latency`：各具体业务逻辑插件在处理链中的实际执行开销耗时分布直方图。

#### L4 Stream 阶段监控指标
- `edge_stream_connection_total`：每个 L4 代理路由的累计连接次数计数。

## proxy_cache 插件使用文档

### 1. 插件概览

`proxy_cache` 插件为网关提供**响应缓存**能力，支持将上游的 HTTP 响应缓存在本地磁盘或共享内存中，并在后续相同请求命中时直接返回缓存内容，从而大幅减少对上游的透传请求，降低后端压力，提升响应速度。

主要应用场景：
- **静态资源加速**：对图片、JS、CSS 等内容进行本地磁盘缓存，避免重复向后端拉取。
- **热点 API 缓存**：对高频读接口的响应进行内存缓存，实现毫秒级命中。
- **降低上游负载**：在流量高峰期，缓存层吸收大量重复请求，保障上游服务稳定。

---

### 2. 配置详情

#### 2.1 插件静态属性 (Attr Config)

通过 `edge.env` 中的 `plugin_attr.proxy_cache` 配置缓存区域（Zone），可定义多个磁盘或内存缓存区。

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |
| `valid` | String / Integer | `"10s"` | 默认全局缓存有效时长。如果是整数，最小值为 `1`，单位为秒；如果是字符串，支持带有时间单位的格式，具体见下方**说明**。 | `"10s"` |
| `zones` | Array\<Object\> | 无 | 缓存区域列表，每个元素定义一个命名缓存区。必须至少包含 1 项，且每项必须唯一。 | 见下方说明 |

> **说明**
>
> *   **`valid` 属性的字符串时间单位格式**：
>     *   秒：`s`（例如：`30s` 表示 30 秒）
>     *   分钟：`m`（例如：`5m` 表示 5 分钟）
>     *   小时：`h`（例如：`2h` 表示 2 小时）
>     *   天：`d`（例如：`6d` 表示 6 天）
> 
> *   **`zones` 数组中每个元素为一个缓存区定义，其字段如下**：
>
>     | 子字段 | 类型 | 描述 |
>     | :--- | :--- | :--- |
>     | `name` | String | 缓存区域名称（必填），对应基础属性中的 `cache_zone`。 |
>     | `memory_size` | String / Integer | 内存共享字典大小（支持单位见下方大小说明）。与 `disk_path` 二选一；设置此项则为内存缓存区。 |
>     | `disk_path` | String | 磁盘缓存目录路径（如 `"/tmp/edge_cache"`）。与 `memory_size` 二选一；设置此项则为磁盘缓存区。 |
>     | `disk_size` | String / Integer | 磁盘缓存最大容量（支持单位见下方大小说明），仅在 `disk_path` 模式下有效。 |
>     | `disk_levels` | String | 磁盘目录层级结构（如 `"1:2"`），仅在 `disk_path` 模式下有效。 |
>
> *   **存储大小单位限制说明（适用于 `memory_size` 与 `disk_size`）**：
>     *   字节：默认单位。如果只写纯数字（如 `512`），则默认表示 512 字节。
>     *   千字节：`k` 或 `K`（例如：`10k` 或 `10K`）
>     *   兆字节：`m` 或 `M`（例如：`10m` 或 `10M`）
>     *   千兆字节：`g` 或 `G`（例如：`10g` 或 `10G`）

`edge.env` 配置示例：

```yaml
plugin_attr:
  proxy_cache:
    zones:
      - name: edge_proxy_cache1
        disk_path: /tmp/edge_disk_cache
        disk_size: 1G
        disk_levels: "1:2"
      - name: memory_cache
        memory_size: 50m
```

---

#### 2.2 插件数据属性 (Metadata Config)

无。

---

#### 2.3 插件基础属性 (Config)

路由级配置，在路由的 `plugins` 节点中配置 `proxy_cache` 对象。

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |
| `cache_type` | String | `"disk"` | 缓存存储类型，支持 `"disk"`（磁盘缓存）和 `"memory"`（内存缓存）。 | `"memory"` |
| `cache_zone` | String | `"edge_proxy_cache1"` (disk) / `"memory_cache"` (memory) | 缓存区域名称，需与静态属性中定义的 `zones[].name` 对应，且类型须与 `cache_type` 匹配。 | `"edge_proxy_cache1"` |
| `cache_key` | String / Array\<String\> | `"${host},${request_uri}"` | 缓存键，支持单个字符串或最多 8 个字符串组成的数组。支持 Edge 变量插值（如 `${host}`、`${uri}`）。 | `"${host},${uri}"` |
| `cache_method` | Array\<String\> | `["GET", "HEAD"]` | 允许缓存的 HTTP 请求方法，枚举值为 `GET`、`POST`、`HEAD`。 | `["GET", "POST"]` |
| `cache_http_status` | Array\<Integer\> | `[200, 301, 404]` | 允许缓存的上游响应状态码列表，取值范围 `200` ~ `599`。 | `[200, 206]` |
| `cache_valid` | Integer | `10` | 缓存有效时长（秒），最小值为 `1`。 | `60` |
| `cache_control` | Boolean | `false` | 是否遵循上游响应的 `Cache-Control` 头来控制缓存行为（包含 `no-cache`、`no-store`、`private`、`max-age` 等指令）。 | `true` |
| `hide_cache_headers` | Boolean | `false` | 是否在返回给客户端的响应中隐藏 `Cache-Control` 和 `Expires` 头。 | `true` |
| `show_resp_cache_header` | Boolean | `false` | 是否在响应中添加缓存状态标识头（名称由 `header_name` 指定）。 | `true` |
| `header_name` | String | `"X-EDGE-Cache-Status"` | 缓存状态响应头名称，仅在 `show_resp_cache_header` 为 `true` 时生效。 | `"X-Cache-Status"` |
| `no_cache_expr` | Array | 无 | 条件表达式，命中时跳过缓存存储（不写缓存）。语法参见 [Edge 表达式列表](使用手册.md#8-核心表达式引擎)。 | `[["arg_nocache", "==", "1"]]` |
| `bypass_cache_expr` | Array | 无 | 条件表达式，命中时绕过缓存读取（直接穿透到上游）。语法参见 [Edge 表达式列表](使用手册.md#8-核心表达式引擎)。 | `[["http_pragma", "==", "no-cache"]]` |

> **关于 `cache_key` 变量支持**
>
> `cache_key` 字段支持 Edge 变量插值，常用变量如 `${host}`、`${uri}`、`${request_uri}`、`${args}` 等，完整列表参见 [Edge 变量列表](使用手册.md#9-内置变量说明)。

> **关于条件表达式**
>
> `no_cache_expr` 和 `bypass_cache_expr` 均使用标准 Edge 表达式格式，示例：`[ ["arg_nocache", "==", "1"] ]`。
> 详见 [Edge 表达式列表](使用手册.md#8-核心表达式引擎)。

## proxy_mirror 插件使用文档

### 1. 插件概览

`proxy_mirror` 插件用于将进入网关的请求**异步镜像**至一个指定的后端地址，同时不影响原始请求的正常代理与响应。常见应用场景包括：

- **流量录制与回放**：将线上流量复制到测试/预发环境，用于问题复现与压测。
- **灰度验证**：将真实流量同步发送至新版本服务，观察其行为而不影响线上用户。
- **日志镜像**：将请求发送至独立的日志收集服务，进行全量审计或分析。

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |
| `host` | String | 无（必填） | 镜像目标地址，格式为 `scheme://host[:port]`，支持 `http`、`https`、`grpc`、`grpcs` 协议。 | `"http://mirror.internal:8080"` |

## proxy_rewrite 使用文档

### 1. 插件概览

`proxy_rewrite` 插件用于在 API 网关将请求代理转发给后端（上游）服务之前，动态修改或重写客户端的请求内容。典型的应用场景包括：
- 动态更改代理请求的 HTTP 方法（如 `GET`, `POST` 等）。
- 更改转发时的请求 Scheme（协议）为 `http` 或 `https`。
- 修改转发的目标 Host 标头。
- 修改请求的 URI 路径，支持正则匹配替换或添加路径前缀。
- 在请求头中添加、重写或通过正则匹配替换标头信息。
- 重写或通过正则匹配替换请求体（Request Body）内容。
- 支持条件表达式以精细化控制何时应用请求头或请求体的修改。

### 2. 配置详情

#### 2.1 插件静态属性 (Attr Config)

无。

#### 2.2 插件数据属性 (Metadata Config)

无。

#### 2.3 插件基础属性 (Config)

路由级配置，可直接配置在特定路由（`Route`）中。支持根据 HTTP 请求方法或基于表达式来进行匹配重写。

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |
| `method` | String | 无 | 重写后的目标请求方法。如 `"POST"`, `"GET"`。 | `"POST"` |
| `scheme` | String | 无 | 重写后的 Scheme，支持 `"http"` 或 `"https"`。 | `"https"` |
| `host` | String | 无 | 重写后的请求 Host。支持带端口。 | `"backend.service:8080"` |
| `uri` | String | 无 | 重写后的请求 URI 路径。支持变量插值解析。 | `"/new-api/v1/${uri}"` |
| `regex_uri` | Array | 无 | 包含 2 个字符串的数组：`[ "匹配正则", "替换值" ]`，用于在原始 URI 上执行正则重写。 | `["^/old-prefix/(.*)", "/new-prefix/$1"]` |
| `add_headers` | Object | 无 | 在原有请求头基础上追加请求头信息。支持变量动态生成。键为 header 名，值支持 String/Number 或复杂结构。 | `{"X-Gateway-Name": "Edge-Prod"}` |
| `headers` | Object | 无 | 覆盖或修改现有的请求头信息。支持通过正则替换或变量动态生成。键为 header 名，值支持 String/Number 或复杂结构。 | `{"Host": "service.internal"}` |
| `body` | String/Array/Object | 无 | 替换原始客户端请求体的内容。当 `plain_text` 为 `false` 时，支持变量插值。 | `"New Body Content"` |
| `regex_body` | Array | 无 | 包含用于在原始请求体上执行正则匹配与替换的规则列表。 | `[["World", "Edge", 1]]` |
| `plain_text` | Boolean | `false` | 设置为 `true` 时，表示 `body` 内容为纯文本，不对其中的任何变量进行解析。 | `true` |
| `include_add_headers_expr` | Array | 无 | 条件表达式规则，仅在满足该条件时执行 `add_headers`。 | `[ ["uri", "==", "/test"] ]` |
| `include_headers_expr` | Array | 无 | 条件表达式规则，仅在满足该条件时执行 `headers` 修改。 | `[ ["uri", "==", "/test"] ]` |
| `include_body_expr` | Array | 无 | 条件表达式规则，仅在满足该条件时执行 `body` 和 `regex_body` 替换。 | `[ ["uri", "==", "/test"] ]` |

> **说明**:
> - `regex_body` 的规则列表每一项的格式为 `[ "正则表达式", "替换字符串", "替换次数" ]`。其中，“替换次数”默认值为 `0`（表示全部替换）。注意：第四个参数“正则选项”为保留选项，不开放给普通用户使用。
> - `headers` 与 `add_headers` 的值除了支持简单的 String/Number 外，还支持正则匹配的高级格式（同样隐藏了保留的正则选项参数）：
>   - 格式一：`[ "提取值", "正则表达式", "替换字符串", "替换次数" ]`（如 `["", "World", "Edge", 1]`），当提取值为空字符串 `""` 时，表示在原 Header 值上执行正则替换。
>   - 格式二：`[ "提取值", 正则规则数组 ]`，用于对同一个源值运行多重正则替换规则。
>   - 格式三：直接传递正则规则数组，在原始 Header 值上做多重正则替换。
> - 条件表达式（如 `include_*_expr` 等）设置示例必须严格使用标准的 Edge 表达式格式，其语法及可用变量详见 [Edge 表达式列表](使用手册.md#8-核心表达式引擎)。

#### 2.4 注意事项

- **内置变量解析**：重写配置（包括 `uri`, `headers`, `add_headers` 以及 `body` 字段）支持注入 Nginx/Edge 内置变量，如 `${host}` 或 `${remote_addr}`，详情见 [Edge 变量列表](使用手册.md#9-内置变量说明)。请注意，`plain_text` 字段仅对 `body` 字段生效（设置为 `true` 时，不对 `body` 中的变量进行解析；而 `uri`、`headers`、`add_headers` 则不受其影响，始终会执行变量解析）。


### 3. 调用与验证示例

以下示例演示如何调用管理接口，配置并应用 `proxy_rewrite` 插件。

> **注意事项**：实际调用管理接口时，请求体需要先经过 SM4 加密，返回结果也需要进行解密，详见 [管理接口调用规范](使用手册.md#3-管理接口调用)。以下示例统一使用大写形式的占位符（如 `API_KEY`）指代敏感信息，占位符未包裹中括号以防歧义。

#### 3.1 覆盖代理 Host 和请求 URI

通过管理接口为指定路由配置插件。将所有请求代理转发到后端的 Host 改写为 `upstream.service.local`，并且重写请求 URI 以追加版本号前缀 `/v2`，同时在请求中注入客户端的真实 IP。

**通过管理接口配置路由**：
```bash

-H 'X-API-KEY: API_KEY' \
-H 'Content-Type: application/json' \
-d '{
    "uri": "/api/*",
    "plugins": {
        "proxy_rewrite": {
            "host": "upstream.service.local",
            "uri": "/v2${uri}",
            "headers": {
                "X-Real-IP": "${remote_addr}"
            }
        }
    },
    "upstream": {
        "nodes": {
            "127.0.0.1:8111": 1
        }
    }
}'
```

**发起客户端请求进行验证**：
```bash
## 发起请求
curl -i http://localhost:9980/api/users

## 后端（127.0.0.1:8111）实际收到的重写后请求为：
## Path: /v2/api/users
## Host Header: upstream.service.local
## X-Real-IP Header: 127.0.0.1
```

#### 3.2 使用条件表达式替换请求体中的关键字

对特定的请求执行请求体的正则查找与替换。当请求 URI 为 `/submit` 时，将请求体中的敏感词 `password` 替换为 `******`。

**通过管理接口配置路由**：
```bash
curl -L -X PUT 'http://ADMIN_ADDR/edge/admin/routes/2002' \
-H 'X-API-KEY: API_KEY' \
-H 'Content-Type: application/json' \
-d '{
    "uri": "/submit",
    "plugins": {
        "proxy_rewrite": {
            "regex_body": [
                [
                    "password",
                    "******",
                    0
                ]
            ],
            "include_body_expr": [
                ["uri", "==", "/submit"]
            ]
        }
    },
    "upstream": {
        "nodes": {
            "127.0.0.1:8111": 1
        }
    }
}'
```

**发起客户端请求进行验证**：
```bash
## 客户端发送包含敏感数据的 POST 请求
curl -i -X POST http://localhost:9980/submit \
-H "Content-Type: text/plain" \
-d "user=admin&password=my_secret_password"

## 后端（127.0.0.1:8111）实际收到的重写后请求体数据为：
## user=admin&******=my_secret_password
```

## real_ip 插件使用说明书

### 1. 插件概览

`real_ip` 插件用于在 API 网关接收到请求时，从请求头（如 `X-Forwarded-For`、`X-Real-IP`）或自定义变量中提取真实的客户端 IP 地址和源端口号，并覆盖重写网关连接的底层源 IP（即更新内置变量 `$remote_addr` 和 `$remote_port`）。

主要应用场景：
- **安全检查与限速**：网关前置了 CDN、高防 WAF 或 L4/L7 负载均衡器时，确保 IP 访问控制（IP 黑白名单）和限速插件能够拿到真正的客户端真实 IP，而不是代理节点的 IP。
- **精确日志记录**：使网关本地日志和上游后端接收到的 `remote_addr` 均更新为客户端的实际 IP，便于追踪分析。

---

### 2. 配置详情

#### 2.1 插件静态属性 (Attr Config)

无。

---

#### 2.2 插件数据属性 (Metadata Config)

通过管理接口进行全局配置（API 路径：`/edge/admin/plugin_metadata/real_ip`），用于设定全局默认的 IP 重写规则。

> **[!IMPORTANT]**
> **所有插件的数据属性（Metadata Config）变更后，必须执行一次插件热加载操作以使配置生效：**
> - **请求方法**：`PUT`
> - **请求路径**：`/edge/admin/plugins/reload`
> 
> *您可以在配置多个插件的数据属性后，统一执行一次热加载操作。*

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |
| `source` | String | `"${http_x_forwarded_for}"` | 提取客户端真实 IP 地址的数据源，支持变量插值（例如通过 `${http_header_name}` 读取指定的标头）。至少包含 1 个字符。 | `"${http_x_real_ip}"` |
| `trusted` | Array | 无 | 受信任的代理服务器 IP 或网段（CIDR 格式）列表。若直接连接的客户端 IP 属于该列表，网关才会信任并解析数据源；若不配置，网关无条件信任数据源并进行解析。 | `["10.0.0.0/8", "192.168.1.1"]` |
| `recursive` | Boolean | `false` | 是否开启递归解析模式。若为 `true`，网关会与 `trusted` 配合从右往左依次遍历代理 IP 列表，剔除属于可信白名单的代理 IP，直至遇到第一个不属于白名单的 IP 作为真实的客户端 IP；若为 `false`，则直接将数据源中最右侧的 IP 作为真实的客户端 IP。 | `true` |

---

#### 2.3 插件基础属性 (Config)

路由级配置，可覆盖全局定义的插件数据属性 (Metadata Config)。

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |
| `source` | String | `"${http_x_forwarded_for}"` | 同上 | `"${http_x_forwarded_for}"` |
| `trusted` | Array | 无 | 同上 | `["192.168.1.0/24", "127.0.0.1"]` |
| `recursive` | Boolean | `false` | 同上 | `true` |

---

### 3. 调用与验证示例

#### 3.1 创建并启用路由

使用 Admin 管理接口创建或更新路由，为路径 `/api/ip` 开启 `real_ip` 插件，将其配置为：仅信任来自本地回环地址 `127.0.0.1` 发起的代理请求，并通过 `X-Forwarded-For` 头部递归获取客户端的真实 IP。

> [!IMPORTANT]
> 调用示例中不包含真实敏感密钥，请将占位符 `API_KEY` 替换为您实际环境中的 Admin API Key。

```bash
curl -i -X PUT \
  -H "Content-Type: application/json" \
  -H "X-Api-Key: API_KEY" \
  -d '{
    "uri": "/api/ip",
    "plugins": {
      "real_ip": {
        "source": "${http_x_forwarded_for}",
        "trusted": ["127.0.0.1"],
        "recursive": true
      }
    },
    "upstream": {
      "nodes": {
        "127.0.0.1:8811": 1
      },
      "type": "roundrobin"
    }
  }' \
  http://127.0.0.1:9991/edge/admin/routes/1002
```

#### 3.2 访问网关进行验证

向网关发起请求，并通过 `X-Forwarded-For` 头传递多级代理链（例如 `1.2.3.4, 127.0.0.1`）。

```bash
curl -i -H "X-Forwarded-For: 1.2.3.4, 127.0.0.1" http://127.0.0.1:9981/api/ip
```

**解析过程**：
1. 网关直接连接源 IP 为 `127.0.0.1`，匹配 `trusted` 列表。
2. 从右往左遍历 `X-Forwarded-For`（即 `127.0.0.1` -> `1.2.3.4`）。
3. `127.0.0.1` 属于可信代理，被剔除。
4. `1.2.3.4` 不属于可信代理白名单，匹配终止，网关最终将 `$remote_addr` 重写为 `1.2.3.4`。

**预期响应输出 (Backend 收到重写后的 $remote_addr)**：
后端服务器收到的客户端来源 IP 地址将被成功改写为 `1.2.3.4`。

## redirect 插件使用文档

### 1. 插件概览

`redirect` 插件用于在网关层实现 HTTP 重定向跳转。用户可以配置静态目标地址、包含动态变量（如 `${host}`, `${request_uri}`）的跳转地址，或利用正则表达式进行匹配与替换，同时支持将 HTTP 请求强制升级为 HTTPS。

主要应用场景：
- **站点迁移/改版跳转**：永久或临时将某个 URL 或路径路由重定向到新地址。
- **强制 HTTPS 跳转**：全站或特定路由强制 HTTP 流量重定向到 HTTPS，保障通信安全。

---

### 2. 配置详情

#### 2.1 插件静态属性 (Attr Config)

无。

---

#### 2.2 插件数据属性 (Metadata Config)

无。

---

#### 2.3 插件基础属性 (Config)

路由级配置。

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |
| `uri` | String | 无 | 重定向的目标 URL/URI，支持动态变量替换。与 `regex_uri` 和 `http_to_https` 互斥。 | `"https://${host}${request_uri}"` |
| `regex_uri` | Array of String | 无 | 包含两个元素的数组，分别代表匹配正则表达式及替换规则。与 `uri` 和 `http_to_https` 互斥。 | `["^/old-api/(.*)", "/new-api/$1"]` |
| `http_to_https` | Boolean | 无 | 是否将 HTTP 请求跳转到 HTTPS。与 `uri` 和 `regex_uri` 互斥。 | `true` |
| `status` | Integer | `302` | 自定义重定向响应的状态码，取值范围为 200 到 599。 | `301` |
| `append_query_string` | Boolean | `true` | 重定向时是否默认附加原始请求的 Query 参数。 | `false` |
| `encode_uri` | Boolean | `false` | 是否对重定向的 URI 部分进行 URL 编码。 | `true` |

> - 配置时必须提供 `uri`、`regex_uri` 或 `http_to_https` 其中之一。

## remote_redis 插件使用文档

### 1. 插件概览

`remote_redis` 插件用于在网关层实时连接 Redis，将查询得到的值自动映射为自定义的网关变量。这可以让后续执行的其他插件（例如 `response_rewrite` 修改响应头、`proxy_rewrite` 重写上游请求头等）能够自由引用该变量值，实现动态的数据传递。

主要应用场景：
- **动态变量提取与重写**：去 Redis 中根据用户 ID 查询对应的详细属性信息，并将其自动填入代理转发给后端服务器的 Header 头部。
- **自定义响应信息**：将查询的缓存值直接渲染进响应 Header 中返回给客户端。

---

#### 2.1 插件数据属性 (Metadata Config)

> **[!IMPORTANT]

##### 2.1.1 全局基础配置

| `key_prefix` | String | `""` | 拼接至 Redis 查询 Key 之前的前缀。 | `""` |

## response_rewrite

### 1. 插件概览

`response_rewrite` 插件用于在 API 网关将响应发送给客户端之前，动态重写响应报文的内容。典型的应用场景包括：
- 修改或覆盖后端的响应状态码（HTTP Status）。
- 动态添加、修改或删除响应头（Response Headers），例如添加安全标头或跨域配置。
- 修改或替换响应体（Response Body），支持纯文本替换、变量注入以及基于正则表达式的内容替换。
- 能够依据条件表达式（Expressions）针对特定的响应内容执行重写操作。

### 2. 配置详情

#### 2.1 插件静态属性 (Attr Config)

无。

#### 2.2 插件数据属性 (Metadata Config)

通过管理接口全局配置（路径：`/edge/admin/plugin_metadata/response_rewrite`）。

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |
| `status` | Integer | 无 | 重写的 HTTP 响应状态码，支持 `200` 到 `599`。 | `200` |
| `add_headers` | Object | 无 | 在原有响应头基础上追加的标头键值对配置。 | `{"X-Add-Header": "Edge"}` |
| `headers` | Object | 无 | 覆盖或设置的响应标头键值对配置（如果原来已有则修改）。 | `{"Server": "Edge-Gateway"}` |
| `body` | String/Array/Object | 无 | 用来替换原始后端响应体的内容。 | `"New Body Content"` |
| `regex_body` | Array | 无 | 包含用于在原始响应体上执行正则匹配与替换的规则列表。 | `[["world", "Edge", 1]]` |
| `plain_text` | Boolean | `false` | 设置为 `true` 时，表示 `body` 内容为纯文本，不对其中的变量进行解析。 | `true` |

> **说明**: `regex_body` 规则列表每一项的格式为 `[ "正则表达式", "替换字符串", "替换次数" ]`。其中，“替换次数”默认值为 `0`（表示全部替换）。

#### 2.3 插件基础属性 (Config)

路由级配置，不仅包含上述数据属性（`_schema`），还额外支持根据 HTTP 请求方法或基于表达式来进行匹配重写。

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |
| `status` | Integer | 同上 | 同上 | 同上 |
| `add_headers` | Object | 同上 | 同上 | 同上 |
| `headers` | Object | 同上 | 同上 | 同上 |
| `body` | String/Array/Object | 同上 | 同上 | 同上 |
| `regex_body` | Array | 同上 | 同上 | 同上 |
| `plain_text` | Boolean | 同上 | 同上 | 同上 |
| `include_add_headers_expr` | Array | 无 | 条件表达式规则，仅在满足该条件时执行 `add_headers`。 | `[ ["status", "==", 200] ]` |
| `include_headers_expr` | Array | 无 | 条件表达式规则，仅在满足该条件时执行 `headers` 修改。 | `[ ["status", "==", 200] ]` |
| `include_body_expr` | Array | 无 | 条件表达式规则，仅在满足该条件时执行 `body` 和 `regex_body` 替换。 | `[ ["status", "==", 200] ]` |
| *(HTTP Method)* | Object | 无 | 可以指定特定的请求方法（如 `"GET"`, `"POST"`）作为键名，值为包含 `status`, `body`, `headers` 等配置的对象，表示仅针对该方法的特定重写规则。 | `{"GET": {"status": 201}}` |

> **说明**：条件表达式（如 `include_*_expr` 等）设置示例必须严格使用标准的 Edge 表达式格式，其语法及可用变量详见 [Edge 表达式列表](使用手册.md#8-核心表达式引擎)。

#### 2.4 注意事项

- **动态变量支持**: 在 `add_headers`、`headers` 的值中，以及 `body` 字段（当 `plain_text` 不为 `true` 时）支持注入 Nginx/Edge 变量，如 `${host}` 或 `${remote_addr}`，请参考 [Edge 变量列表](使用手册.md#9-内置变量说明)。
- **响应体过滤限制**: 使用 `body` 或 `regex_body` 修改响应体时，如果后端的响应是流式的或被分块传输（Chunked），网关会先缓冲响应体内容再执行替换，可能会增加一定的内存消耗或请求延迟。
- **配置覆盖**: 路由级别的配置与基于 HTTP 方法（如 `GET`）的配置如果存在冲突，以具体方法块内的配置为准。

## security_common_args 插件使用文档

## security_corerule

### 1. 插件概览

`security_corerule` 插件是一个基于 OWASP Core Rule Set (CRS) 的安全检查引擎，用于拦截恶意的 HTTP 请求（如 SQL 注入、跨站脚本攻击、命令注入等）。其典型应用场景包括：
- 启用基础 Web 应用防火墙（WAF）来保护后端服务。
- 针对特殊场景在路由级别上配置白名单和黑名单，跳过特定的请求检查或强制进行拦截。
- 指定针对 URI、参数、Cookie 和请求体的解析方式，以适应特定业务场景的数据格式。

## traceid 插件使用说明书

### 1. 插件概览

`traceid` 插件用于在网关层为每个请求自动生成或传递一个全局唯一的追踪 ID（Trace ID），并将其注入至请求头中转发给上游后端服务。该 ID 还可以选择性地回显在响应头中返回给客户端。

主要应用场景：
- **全链路追踪**：作为系统入口，为请求分配唯一的 Trace ID，方便与上游微服务、数据库日志、调用链追踪系统（如 Zipkin, Jaeger 等）打通。
- **问题快速定位**：当客户端请求出错时，可以通过响应头中回显的 Trace ID 快速在网关及后端的日志中检索、定位根因。

---

### 2. 配置详情

#### 2.1 插件静态属性 (Attr Config)

无。

---

#### 2.2 插件数据属性 (Metadata Config)

通过管理接口进行全局配置（API 路径：`/edge/admin/plugin_metadata/traceid`），用于设定全局默认的追踪行为。

> **[!IMPORTANT]**
> **所有插件的数据属性（Metadata Config）变更后，必须执行一次插件热加载操作以使配置生效：**
> - **请求方法**：`PUT`
> - **请求路径**：`/edge/admin/plugins/reload`
> 
> *您可以在配置多个插件的数据属性后，统一执行一次热加载操作。*

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |
| `header_name` | String | `"X-EDGE-TraceID"` | 追踪标头名。长度限制在 1 至 64 个字符。 | `"X-Custom-Trace-ID"` |
| `ignore_header` | Boolean | `false` | 是否忽略客户端请求中已携带的同名 Trace ID 头字段。若为 `true`，网关会无条件忽略客户端传入的值并强制生成新 Trace ID；若为 `false`，则优先继承客户端的值。 | `true` |
| `show_resp_header` | Boolean | `false` | 是否在响应给客户端的 Header 中回显追踪 ID。 | `true` |

---

#### 2.3 插件基础属性 (Config)

路由级配置，可覆盖全局定义的插件数据属性 (Metadata Config)。

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |
| `header_name` | String | `"X-EDGE-TraceID"` | 同上 | `"X-Route-Trace-ID"` |
| `ignore_header` | Boolean | `false` | 同上 | `true` |
| `show_resp_header` | Boolean | `false` | 同上 | `true` |

---

### 3. 调用与验证示例

#### 3.1 创建并启用路由

使用 Admin 管理接口创建或更新路由，为路径 `/api/trace` 开启 `traceid` 插件，将其配置为：自定义标头 `X-My-Trace-ID` 并在响应头中回显。

> [!IMPORTANT]
> 调用示例中不包含真实敏感密钥，请将占位符 `API_KEY` 替换为您实际环境中的 Admin API Key。

```bash

  -H "Content-Type: application/json" \
  -H "X-Api-Key: API_KEY" \
  -d '{
    "uri": "/api/trace",
    "plugins": {
      "traceid": {
        "header_name": "X-My-Trace-ID",
        "ignore_header": true,
        "show_resp_header": true
      }
    },
    "upstream": {
      "nodes": {
        "127.0.0.1:8811": 1
      },
      "type": "roundrobin"
    }
  }' \
  http://127.0.0.1:9991/edge/admin/routes/1001
```

```bash

```

**预期响应输出 (部分)**：

## tracing 插件使用文档

### 1. 插件概览

`tracing` 插件用于在网关层动态修改、新增或覆盖请求和响应的 HTTP Header。用户可使用系统变量（如客户端真实 IP、随机 UUID 等）组装追踪标识并将其透传至上游服务或客户端，从而实现全链路的可观测性监控。

主要应用场景：
- **请求链路 Trace ID 注入**：自动向经过网关的 HTTP 请求注入唯一的 Trace ID 标头，方便分布式微服务架构中统一进行日志抓取与追踪。
- **获取并透传网络元数据**：提取当前请求的客户端真实地址及地理位置信息并封装入 Header 透传。

---

### 2. 配置详情

#### 2.1 插件静态属性 (Attr Config)

无。

---

#### 2.2 插件数据属性 (Metadata Config)

无。

---

#### 2.3 插件基础属性 (Config)

路由级配置。

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |
| `bypass_missing_key` | Boolean | `true` | 若解析变量的值为空时，是否直接忽略该标头注入。 | `true` |
| `bypass_error_key` | Boolean | `true` | 解析变量出错时是否忽略。 | `true` |
| `header_name` | String | `"X-EDGE-Trace-Info"` | 触发 `show_resp_header` 时，用于在响应中携带已追踪修改之标头列表的 Header 名称。 | `"X-Trace-Info"` |
| `show_resp_header` | Boolean | `false` | 是否在 HTTP 响应头中添加追踪指示 Header，表明哪些追踪头已被网关设置或覆盖。 | `true` |
| `headers` | Object | 无 | 需要注入的请求头及其对应的包含变量的格式串映射。最大支持 20 个 Key。 | `{"X-Request-Id": "${uuid}"}` |
| `replace_header` | Boolean | `false` | 是否替换覆盖请求中已存在的同名请求头。 | `true` |
| `resp_headers` | Object | 无 | 需要注入的响应头及其对应的包含变量的格式串映射。最大支持 20 个 Key。 | `{"X-Trace-Response-Time": "${upstream_response_time}"}` |
| `replace_resp_header` | Boolean | `false` | 是否替换覆盖响应中已存在的同名响应头。 | `true` |

---

### 3. 调用与验证示例

本示例介绍如何创建路由并启用 `tracing` 插件，在发送给上游的请求中注入唯一 Trace ID，在返回的响应中注入上游处理耗时。

> [!IMPORTANT]
> 调用示例中不包含真实敏感密钥，请将占位符 `API_KEY` 替换为您实际环境中的 Admin API Key。

#### 3.1 步骤 1：创建启用 tracing 插件的路由

创建路由关联路径 `/test_trace`，并在插件配置中定义 `headers` 和 `resp_headers`：

```bash
curl -i -X PUT \
  -H "Content-Type: application/json" \
  -H "X-Api-Key: API_KEY" \
  -d '{
    "uri": "/test_trace",
    "plugins": {
      "tracing": {
        "headers": {
          "X-Trace-Id": "${uuid}"
        },
        "replace_header": true,
        "resp_headers": {
          "X-Process-Time": "${request_time}"
        },
        "replace_resp_header": true,
        "show_resp_header": true
      }
    },
    "upstream": {
      "nodes": {
        "127.0.0.1:8811": 1
      },
      "type": "roundrobin"
    }
  }' \
  http://127.0.0.1:9991/edge/admin/routes/3004
```

---

#### 3.2 访问网关进行验证

向网关发起访问请求：

```bash
curl -i http://127.0.0.1:9981/test_trace
```

**预期响应输出（响应头中成功包含指定的追踪头信息）**：

```http
HTTP/1.1 200 OK
Content-Type: text/plain
Connection: keep-alive
X-Process-Time: 0.003
X-EDGE-Trace-Info: |X-Process-Time
...
```
*(提示：由于配置了 `"show_resp_header": true`，响应头中额外附带了 `X-EDGE-Trace-Info`，其值为 `|X-Process-Time`，指示响应中的 `X-Process-Time` 头是通过 tracing 插件生成并写入的。而在转发到后端上游的请求包中，已附带了包含随机 UUID 的 `X-Trace-Id` 头)*

## traffic_limit_count 使用文档

### 插件概览

`traffic_limit_count` 插件是网关提供的限流插件，支持在特定的时间窗口内限制指定变量（如客户端 IP、请求 URI 等）的累计访问次数。支持本地单机限流（性能最高）以及结合 Redis 的分布式多节点共享限流。

---

### 配置详情

#### 1. 插件静态属性 (Attr Config)

该属性用于在项目根目录的 `edge.env` 文件下的 `plugin_attr.traffic_limit_count` 块中为当前节点单独配置限制倍率。

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |
| `local_ratio` | Number | 无 | 针对 `local` 本地限流策略特有的限流倍率，必须大于 `0`。在 `local` 策略下，允许的请求数调整为 `count * local_ratio` 并向下取整，最小为 `1`。 | `1.5` |

---

#### 2. 插件数据属性 (Metadata Config)

通过管理节点接口进行全局配置（路径：`/edge/admin/plugin_metadata/traffic_limit_count`）。

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |
| `key` | String | `$` | 默认的限流键，支持网关内置变量（如 `${remote_addr}`）。 | `"${remote_addr}"` |
| `status` | Integer | `403` | 超出限制被拦截时，默认返回的 HTTP 状态码。 | `429` |
| `message` | String | 无 | 被限流拦截时，默认返回的响应消息。 | `"Too Many Requests"` |
| `bypass_missing_key` | Boolean | `true` | 当无法解析提取限流键时是否直接放行。 | `false` |
| `bypass_error_limit` | Boolean | `true` | 当执行限流逻辑出错（如 Redis 连接断开）时是否放行。 | `false` |
| `header_name` | String | `"X-EDGE-LimitCount-Remaining"` | 展示剩余次数的自定义响应头名称。 | `"X-RateLimit-Remaining"` |
| `show_resp_limit_header` | Boolean | `false` | 是否向响应中添加包含剩余请求次数的头部信息。 | `true` |
| `redis_dc` | String | `"TEST"` | 默认的 Redis 数据中心标识名称。 | `"IDC-1"` |
| `redis_conf` | Object | 无 | Redis 连接详细配置。 | 见下文示例 |

> **说明**: 关于 `redis_conf` 对象内部配置属性的详细说明：
> - **主要子属性**：
>   - `MODE`：运行模式，Enum `["redis", "rediscluster", "proxycluster"]`。
>   - `AUTH`：连接鉴权密码。
>   - `DATABASE`：数据库索引（单机模式），默认 `0`。
>   - `BACKLOG`：连接池的积压（backlog）大小，默认 `-1`（表示不启用）。
>   - `TIMEOUT`：网络超时时间（单位秒），默认 `1`。
>   - `MAX_CONNECT_NODES`：每个 worker 最大连接的节点数，默认 `2`，配置为 `0` 时表示不限制。
>   - `MAX_IDLE_TIME`：连接最大空闲时间（单位秒），默认 `60`。
>   - `CONNECTION_MAX_SIZE`：连接池大小，默认 `1024`。
>   - `CLUSTER_{cluster_name}`：各数据中心（集群）对应的 Redis 节点地址列表。其值为包含节点配置子数组的列表（格式如 `[ [ "ip/host", port, priority_or_group, weight, recovery_time_seconds ], ... ]`）。例如：`"CLUSTER_TEST": [ ["127.0.0.1", 6379, 0, 10, 1000] ]`。
> - **属性值的表示形式**：除了 `CLUSTER_{cluster_name}` 之外，其他的 Redis 配置属性在 `redis_conf` 中均支持以下两种表示方式：
>   1. **对象映射方式（Object mapping）**：以属性名作为顶层 Key，其值是一个以集群名称为 Key 的 JSON 对象。例如：
>      `"MODE": { "TEST": "redis", "IDC-1": "rediscluster" }`
>   2. **后缀字段名方式（Suffix keys）**：在属性名后加上下划线和对应的集群名称作为顶层 Key。例如：
>      `"MODE_TEST": "redis"`, `"MODE_IDC-1": "rediscluster"`
>   - **优先级**：**对象映射方式**的优先级高于**后缀字段名方式**。

---

#### 3. 插件基础属性 (Config)

路由级属性，启用该插件时配置在路由下。

| 字段名 | 类型 | 默认值 | 描述 | 设置示例 |
| :--- | :--- | :--- | :--- | :--- |
| `status` | Integer | 同上 | 覆盖默认的限流 HTTP 状态码。 | `429` |
| `message` | String | 同上 | 覆盖默认的被拦截响应信息。 | `"Limits Exceeded"` |
| `bypass_missing_key` | Boolean | 同上 | 无法提取限流键时是否放行。 | `true` |
| `bypass_error_limit` | Boolean | 同上 | 限流计算出错时是否放行。 | `true` |
| `header_name` | String | 同上 | 覆盖默认的剩余次数响应头名称。 | `"X-RateLimit-Remaining"` |
| `show_resp_limit_header` | Boolean | 同上 | 是否在响应体中包含剩余请求次数。 | `true` |
| `redis_dc` | String | 同上 | 覆盖 Redis 数据中心名称。 | `"IDC-2"` |
| `redis_conf` | Object | 同上 | 覆盖 Redis 连接配置。 | 见下文示例 |
| `limits` | Array | 见下文 | 限流规则列表。 | 见下文示例 |
| `policy` | String | `"local"` | 限流存储策略，Enum `["local", "redis"]`。 | `"local"` |
| `group` | String | 无 | 限流组名称，同一组内的不同路由会共享计数。 | `"api_rate_limiter"` |

> **说明**: `limits` 内每个限制项包含以下三个字段：
> - `key`：限流的特征键，支持动态变量（如 `${uri}`, `${host}`），默认值为 `$`。当设置为 `$` 时，表示限流针对的是当前路由或接口整体，即该路由下的所有请求共享同一个计数器；常用的变量可以参考 [Edge 变量列表](使用手册.md#9-内置变量说明)。
> - `count`：指定窗口内允许的最大请求次数，默认 `3`。
> - `window`：限流的时间窗口大小，单位：秒，默认 `5`。

---

#### 1. 全局配置插件数据属性 (Metadata)

首先，我们可以通过管理节点接口配置全局的默认 Redis 连接信息（此处以本地 Redis 为例）：

