## Context

当前上游创建/编辑弹窗（`ClusterUpstreams.vue` + `useClusterUpstreams.ts`）的高级配置 Tab 中，健康检查区域使用一个复选框（ON/OFF）控制启用状态，内容为预设 JSON 文本域 `{"passive": {}, "active": {"unhealthy": {}}}`。用户必须手写 JSON 才能改变参数，对健康检查结构不熟悉的用户完全不知道如何配置。

APISIX 的健康检查 JSON 结构包含 active（主动探测）和 passive（被动观测）两大分支，每分支又有 healthy/unhealthy 子结构，涉及检查类型、路径、间隔、超时、并发、HTTP 状态码、失败阈值等十余个参数。

**约束条件：**
- 保持与现有高级配置项的交互一致性（toggle + 表单控件，toggle OFF 时控件置灰）
- 必须保留 JSON 编辑能力（AGENTS.md 规定"表单编辑和 JSON 编辑都支持，不可移除任一模式"）
- checks 字段在后端存储为 JSON 字符串（`Text` 列），API 层为 `Dict[str, Any]`，数据格式不变

## Goals / Non-Goals

**Goals:**
- 将健康检查 JSON 文本域替换为结构化表单，使用户无需理解 JSON 结构即可完成常用配置
- 提供三种模式快速选择：仅主动检查、仅被动检查、主动+被动检查
- 提供主动/被动各自的参数输入控件（输入框、选择器、数字输入）
- 保留"编辑原始 JSON"弹窗入口，支持表单↔JSON 双向同步
- 保持与现有高级配置一致的 UI 模式和交互风格

**Non-Goals:**
- 不改变后端 checks 字段的数据模型和存储方式（仍是 JSON 字符串）
- 不改变 API 路由和请求/响应格式
- 不引入新的外部依赖
- 不涉及数据库迁移
- 不改变其他高级配置项（超时、连接池、重试等）

## Decisions

### 1. 模式选择：RadioGroup 而非 Select 或独立 Checkbox

**决策**：使用三个 radio button（仅主动 / 仅被动 / 主动+被动），而非两个独立 checkbox（主动/被动）。

**理由**：主动和被动不能同时关闭（既然打开了健康检查 toggle，至少需要一个模式），radio 天然保证"至少选一个"的语义。独立 checkbox 需要额外处理互斥逻辑，且无法表达"一个都不选"的无效状态。

### 2. 表单↔JSON 双向同步

**决策**：表单字段变更时实时同步到内部 `checks` 对象；用户点击"编辑原始 JSON"按钮打开弹窗，弹窗关闭时解析 JSON 回填到表单。

**理由**：
- 表单操作是主要交互方式，实时同步保证提交时数据准确
- JSON 编辑作为进阶入口，只在用户明确需要手动编辑 JSON 时触发回填
- 双向同步能避免"表单改了但 JSON 没更新"或"JSON 改了但表单字段还是旧值"的不一致问题

### 3. 健康判断/不健康判断使用折叠面板（Collapse）

**决策**：主动/被动的内部子区域（健康判断、不健康判断）默认折叠，用户按需展开。

**理由**：大多数用户只需要改间隔和阈值，健康判断和不健康判断的详细参数（HTTP 状态码列表）使用频率较低。默认折叠可以让表单初次看起来更简洁，与现在"高级配置"展开才显示详情的模式一致。

### 4. HTTP 状态码输入使用标签输入组件（Tag Input）

**决策**：HTTP 状态码使用 `a-select mode="tags"` 或 `a-input` 支持逗号分隔输入，而非固定数量输入框。

**理由**：健康/不健康的状态码列表长度不固定，标签输入最灵活。常用状态码可作为预设选项供选择，同时允许用户自由输入。

**附加规则**：
- 每个状态码必须是 100-599 的数字
- 非法值输入时拒绝添加并提示
- 自动去重并升序排列
- 不支持范围简写（如 `500-505`），如有需要请走 JSON 编辑器

### 5. 检查类型为 tcp 时隐藏 HTTP 相关字段

**决策**：当主动或被动检查的 type 选择 "tcp" 时，HTTP 专有字段（http_path、http_statuses、https_verify_certificate）自动隐藏。

**理由**：TCP 健康检查不涉及 HTTP 路径、状态码和证书验证，显示这些字段会造成困惑。字段隐藏——不相关的参数根本不出现。

### 6. 模式切换不重置已填值

**决策**：用户在"仅主动"和"主动+被动"之间切换时，保留已填写的参数值。

**理由**：被动部分的值在切换回"主动+被动"时应该保留，而不是每次重置为默认值。用户的填写内容应该被尊重。仅当新建上游（首次进入表单）时才使用默认值填充。

### 7. 提供"重置为默认"按钮

**决策**：在健康检查表单区域提供"重置为默认"按钮，点击后将当前模式对应的字段恢复为默认值，模式选择保持不变。

**理由**：用户可能在"仅被动"或"主动+被动"模式下调乱了参数，重置不应该改变他选择的模式。只重置当前模式的字段，其他模式的值不受影响。

## Risks / Trade-offs

| 风险 | 缓解措施 |
|---|---|
| 表单字段过多（主动+被动共十余个参数）看上去复杂 | 使用 Collapse 折叠健康/不健康子区域；默认只展开最常用的参数区域 |
| JSON 双向同步可能丢失数据结构（如用户手动写了非标准字段） | JSON 编辑弹窗关闭时，解析 JSON 后与表单字段做 diff，非标准字段保留在 JSON 中不被覆盖 |
| 模式切换时某些字段在 JSON 中出现/消失可能导致验证不一致 | 提交时根据当前模式构建 JSON，已关闭的模式对应的字段不在 JSON 中 |
