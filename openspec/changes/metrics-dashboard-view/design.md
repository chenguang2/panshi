## Context

已有指标监控基础功能：后端 API 和单指标查询页面（`/metrics`）。现有页面需要用户手动选择指标，无法全局纵览。需要在保留单指标查询的同时，新增一个 Grafana 风格的多图仪表盘。

前端已安装 ECharts + vue-echarts，后端 API 支持 `since`/`interval`/`label` 参数，可复用。

## Goals / Non-Goals

**Goals:**
- 新增 `/metrics/dashboard` 页面，以 2 列网格展示多个指标图表
- 侧边栏"指标监控"改为父子菜单结构（"查询分析" + "仪表盘"）
- 页面顶部提供全局时间范围、刷新间隔、自动刷新控制
- 前端并发加载多个指标数据，减少等待时间
- 默认展示业务指标（QPS、活跃连接数、错误率），基础设施指标折叠
- 每个图表支持 tooltip、空数据状态

**Non-Goals:**
- 不改动后端 API（完全复用现有接口）
- 不做图表拖拽排序、自定义布局
- 不做图表的全屏展开
- 不做指标阈值告警叠加

## Decisions

### 1. 并发查询策略
- **选择**：使用 `Promise.allSettled` 并行发起所有指标的查询请求
- **理由**：单个指标查询独立，互不依赖；全部完成后再统一渲染，避免部分图表闪烁
- **备选**：串行查询 → 页面加载慢；逐图表渲染 → 视觉上逐个弹出，体验不一致

### 2. 图表布局
- **选择**：CSS Grid 2 列，每个图表卡片统一高度 280px
- **理由**：2 列在 1200px 及以上分辨率下阅读舒适；固定高度避免页面跳动
- **备选**：3 列 → 图表太窄；flex wrap → 高度不统一

### 3. 折叠区域设计
- **选择**：业务指标默认展开，基础设施指标（scrape_*、up、shared_dict_*）放在折叠区域，点击"更多指标"展开
- **理由**：运维人员大部分时间关注业务指标；基础设施指标在排查问题时才需要
- **备选**：全部展开 → 页面太长；tab 切换 → 增加操作步骤

### 4. 菜单结构调整
- **选择**："指标监控"改为不可点击的父菜单（展开图标），子菜单为"查询分析"和"仪表盘"
- **理由**：Ant Design Vue 的 a-sub-menu 支持此模式
- **备选**：平铺两个菜单项 → 侧边栏条目过多

### 5. 刷新策略
- **选择**：仪表盘使用独立的刷新计时器（可配置间隔 60s/120s/300s，默认 60s），与单指标页面独立
- **理由**：采集周期为 60s，小于 60s 的刷新间隔会看到重复数据；300s 适合长时段观察

### 6. 路由注册方式
- **选择**：在 `setupDynamicRoutes()` 中为 `metrics` 特性添加两个独立路由（`/metrics` 和 `/metrics/dashboard`），而非修改 featureRouteMap 结构
- **理由**：保持现有的 flat featureRouteMap 模式；两个路由共享同一个 feature 开关

### 7. 当前值显示
- **选择**：从 time_series 查询结果中取最后一个 bucket 的数据点，不额外调用 summary API
- **理由**：零额外请求成本，数据与图表一致；间隔非常短（秒级差距），用户无感知

### 8. 并发策略
- **选择**：所有指标请求使用 `Promise.allSettled` 同时并发，不限制并发数
- **理由**：后端单次查询 100-300ms，8 个并发总耗时 ≈ 300ms；通过请求超时（10s）兜底

## Risks / Trade-offs

| 风险 | 缓解措施 |
|---|---|
| **并发请求过多**：同时查 6-8 个指标，ClickHouse 压力增大 | 限制默认并发数；使用 Promise.allSettled 容错 |
| **页面加载慢**：等所有请求完成才渲染 | 设置请求超时（10s）；超时的图表显示加载失败 |
| **图表过多视觉杂乱**：10+ 指标同时展示 | 折叠基础设施指标；默认只展示 3 个关键业务指标 |
| **父子菜单 UI 复杂度**：侧边栏层级增加 | 参考现有菜单样式，保持视觉一致 |
